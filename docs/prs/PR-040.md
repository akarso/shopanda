# PR-040 — Admin Route Group + Role Guard

## Summary

Adds role-based access control so `/admin/*` routes require the `admin` role.
Customers with the `customer` role receive **403 Forbidden**; unauthenticated
requests receive **401 Unauthorized**.

## Changes

### Domain — Customer Role

Added `customer.Role` type (`"customer"` | `"admin"`) to the customer domain,
following the same pattern as `customer.Status`.

- `RoleCustomer` — default for all new registrations
- `RoleAdmin` — grants access to admin endpoints
- `Role.IsValid()` validates known values
- `NewCustomer` defaults `Role` to `RoleCustomer`

### Persistence

- **Migration** `011_add_customer_role.sql` — adds `role TEXT NOT NULL DEFAULT 'customer'` with `CHECK` constraint to the customers table. Existing rows become `'customer'`.
- **Postgres customer repo** — all SELECT, INSERT, UPDATE queries and `scanCustomer` updated to include `role` column with validation.

### Auth Service

`Register` and `Login` now use `c.Role` (from the customer record) instead of
the hardcoded `identity.RoleCustomer` when creating JWT tokens. This means
customers with `role = 'admin'` in the database receive admin tokens on login.

### Route Wiring

`cmd/api/main.go`:
- Added `requireAdmin := shophttp.RequireRole(identity.RoleAdmin)`
- Admin routes changed from `HandleFunc` (unguarded) to `Handle` with `requireAdmin`:

```text
POST   /api/v1/admin/products                  ← requireAdmin
PUT    /api/v1/admin/products/{id}              ← requireAdmin
POST   /api/v1/admin/products/{id}/variants     ← requireAdmin
PUT    /api/v1/admin/products/{id}/variants/{variantId} ← requireAdmin
```

- Public read routes (`GET /products`, `GET /variants`) remain unguarded.

## Design Notes

- `RequireRole(admin)` was already implemented in a prior PR along with `Identity.HasRole()` and test helpers (`AdminRequest`, `CustomerRequest`). This PR wires it to the routes and provides the data model to support it.
- Admin assignment is via direct database update (`UPDATE customers SET role = 'admin' WHERE ...`). No admin self-registration endpoint exists — this is intentional per the principle of explicit, minimal scope.
- `customer.Role` is separate from `identity.Role` to avoid cross-domain coupling. The auth service bridges them by passing `string(c.Role)` to `jwt.Create`, which the token parser reads back as `identity.Role`.
