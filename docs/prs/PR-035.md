# PR-035: Checkout Workflow Engine

## Summary

Introduces the checkout workflow infrastructure — a sequential step-based pipeline engine that coordinates order creation. Defines the `Context` (pass-through object), `Step` interface (unit of work), `Workflow` executor (sequential runner with event emission and execution trace), and `Service` (application-layer orchestrator that hydrates the cart and runs the workflow). No concrete checkout steps are included; those follow in subsequent PRs.

## Changes

### Application Layer

- **`internal/application/checkout/context.go`** *(new)* — `Context` struct carrying `CartID`, `Cart`, `CustomerID`, `Currency`, `Order`, `Meta` (key-value markers for idempotency), and `Trace` (execution log). `NewContext()` constructor. `SetMeta()`/`GetMeta()` accessors.
- **`internal/application/checkout/step.go`** *(new)* — `Step` interface with `Name() string` and `Execute(ctx *Context) error`.
- **`internal/application/checkout/workflow.go`** *(new)* — `Workflow` executor. `NewWorkflow(steps, bus, log)` constructor. `Execute()` runs steps sequentially, emits `checkout.step.started`/`checkout.step.completed` events per step, stops on first error with `checkout.failed` event, emits `checkout.completed` on success. Builds execution `Trace` on the context. Event payload structs: `StepStartedData`, `StepCompletedData`, `CheckoutFailedData`, `CheckoutCompletedData`.
- **`internal/application/checkout/service.go`** *(new)* — `Service` with `NewService(carts, workflow, log)` constructor (nil-guarded). `StartCheckout(ctx, cartID, customerID)` loads cart, validates ownership/active status/non-empty, initialises `Context`, runs workflow. Returns `(*Context, error)` — context is returned even on workflow error so callers can inspect the trace.

### Wiring

- **`cmd/api/main.go`** — `checkoutWorkflow` and `checkoutService` instantiated (no steps registered yet; no HTTP endpoint). Ready for step registration in subsequent PRs.

## Design Decisions

- **Infrastructure-only PR**: No concrete steps (validate_cart, create_order, etc.) — those are separate PRs following the "one PR = one responsibility" rule. This PR delivers only the engine.
- **Sequential execution, no rollback**: Per the checkout spec, steps run in order; failure stops execution. Compensation is handled via events, not automatic rollback.
- **Steps as constructor argument**: Steps are passed to `NewWorkflow()` as an ordered slice rather than using a global registry. This keeps behaviour explicit and testable. A position-based registry can layer on top if needed.
- **Context returned on error**: `StartCheckout` returns the context even when the workflow fails, allowing callers to inspect `Trace` for diagnostics.
- **Idempotency via Meta**: Steps use `ctx.SetMeta()`/`ctx.GetMeta()` to store markers (e.g. `"created_order": true`) so they can be safely re-invoked.
- **No HTTP endpoint yet**: `POST /checkout` will be added in a later PR when concrete steps exist.
