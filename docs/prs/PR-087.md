# PR-087 — Invoice Domain + Storage

## Summary

Introduces the **invoice** bounded context: domain entities (Invoice, CreditNote, Item), repository interfaces, event definitions, PostgreSQL repositories with DB-sequence numbering, and a migration.

## Scope

Per ROADMAP Phase 22 — Invoicing, this PR covers:

- Invoice entity (immutable financial snapshot of an order)
- Credit note entity (immutable correction document linked to an invoice)
- Item value object (shared by invoices and credit notes)
- Sequential numbering via PostgreSQL sequences
- Repository interfaces + Postgres implementations
- Domain events (`invoice.created`, `credit_note.created`)
- Migration 025

## Design Decisions

1. **Immutable documents**: Invoices have no `UpdatedAt`; corrections are modeled as separate CreditNote entities.
2. **DB sequence numbering**: `invoice_number_seq` and `credit_note_number_seq` generate sequential numbers via `DEFAULT nextval(...)`. The number is returned via `RETURNING` and written back to the domain struct after commit.
3. **Shared Item type**: Both Invoice and CreditNote reuse the same `Item` struct (no `CreatedAt` needed since documents are immutable snapshots).
4. **Separate repos**: `InvoiceRepository` and `CreditNoteRepository` are distinct interfaces — credit notes are independent documents with their own numbering, not child records of invoices.
5. **Single currency column**: Each document has one `currency` column; all Money fields (subtotal, tax, total) use it.
