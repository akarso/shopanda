# PR-088 — Invoice Generation + PDF

## Summary

Adds the **invoice generation** use case: snapshot a paid order into an immutable invoice, render a PDF, store it, and publish the `invoice.created` event.

## Scope

Per ROADMAP Phase 22 — Invoicing, this PR covers:

- Application service for order → invoice generation
- PDF renderer port (domain interface)
- fpdf-based PDF renderer infrastructure implementation
- PDF storage via `media.Storage`
- Event publishing (`invoice.created`)

## Design Decisions

1. **PDFRenderer as port**: Defined in the domain layer so the application service depends on an abstraction. The fpdf implementation lives in infrastructure and can be swapped (e.g. for an HTML-to-PDF plugin using headless Chrome).
2. **TaxAmount as input**: Order doesn't carry tax breakdown, so `GenerateInput.TaxAmount` is provided by the caller (tax service, pricing pipeline, or API). This keeps invoice generation decoupled from tax calculation.
3. **Paid-only guard**: Only orders in `paid` status may be invoiced — prevents premature invoice generation.
4. **Idempotency check**: `FindByOrderID` rejects duplicate invoices (backed by the UNIQUE constraint from PR-087).
5. **PDF storage path**: `invoices/<invoice_id>/invoice-<number>.pdf` — deterministic, retrievable via `media.Storage.URL()`.
6. **Event fire-and-forget**: Event publish errors are logged but don't fail the operation (matches existing media service pattern).
