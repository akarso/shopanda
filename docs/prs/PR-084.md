# PR-084 — Rule Primitives

## Summary

Adds a generic, type-safe rule system in `internal/domain/rule/`. Provides
`Condition[T]`, `Rule[T]`, and a sequential `Executor[T]` that will power
promotions (PR-085/086), cart validation, and shipping eligibility.

## Changes

### New files

| File | Purpose |
|------|---------|
| `internal/domain/rule/condition.go` | `Condition[T]` interface + `ConditionFunc[T]` adapter |
| `internal/domain/rule/rule.go` | `Rule[T]` struct (Name, Condition, Apply) |
| `internal/domain/rule/executor.go` | `Executor[T]` — sequential rule runner |
| `internal/domain/rule/rule_test.go` | 14 test cases + 1 example |

## Design

### Condition

```go
type Condition[T any] interface {
    Evaluate(ctx T) bool
}
```

Receives a **value** (read-only view). `ConditionFunc[T]` adapts a plain
function to this interface.

### Rule

```go
type Rule[T any] struct {
    Name      string
    Condition Condition[T]
    Apply     func(ctx *T) error
}
```

Exported fields — no constructor needed. `Apply` receives a **pointer** so it
can mutate the context.

### Executor

```go
exec := rule.NewExecutor(rules...)
err  := exec.Execute(&ctx)
```

Sequential evaluation: for each rule, if `Condition.Evaluate` returns true,
`Apply` is called. First error halts execution. The error wraps the rule name
for diagnostics.

### Key properties

| Property | Guarantee |
|----------|-----------|
| Execution order | Sequential, insertion order |
| Condition sees current state | Yes — re-evaluated per rule after prior mutations |
| Error stops pipeline | Yes — first Apply error halts |
| Error wrapping | `rule "name": underlying error` |

## Test coverage

- `TestConditionFunc_True` / `_False` — adapter works both ways
- `TestRule_AppliesWhenConditionTrue` — basic rule application
- `TestExecutor_Empty` — no rules, no-op
- `TestExecutor_MatchingRule` — single matching rule mutates context
- `TestExecutor_NonMatchingRule` — non-matching rule is skipped
- `TestExecutor_SequentialOrder` — 3 rules execute in order
- `TestExecutor_SkipsNonMatching` — middle rule skipped, others apply
- `TestExecutor_ErrorHaltsExecution` — error stops before third rule
- `TestExecutor_ErrorWrapsRuleName` — error message includes rule name
- `TestExecutor_ConditionSeesCurrentState` — second condition sees first rule's mutation
- `TestExecutor_MultipleRulesMutate` — cumulative mutations
- `TestExecutor_PricingAnalog` — promotion-style threshold discount (sub-tests)
- `ExampleExecutor` — runnable documentation
