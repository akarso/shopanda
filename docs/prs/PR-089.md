# PR-089 — URL Rewrite System

## Phase
Phase 23 — URL Routing & CMS

## Scope
`url_rewrites` table, migration, resolver middleware, catch-all route.

## Files

### Domain
- `internal/domain/routing/rewrite.go` — `URLRewrite` entity (path, type, entity_id) with constructor + validation
- `internal/domain/routing/repository.go` — `RewriteRepository` interface (FindByPath, Save, Delete)
- `internal/domain/routing/context.go` — Context helpers: `WithRewrite`, `RewriteFrom`

### Infrastructure
- `migrations/026_create_url_rewrites.sql` — `url_rewrites` table with `path` PK and `entity_id` index
- `internal/infrastructure/postgres/rewrite_repo.go` — PostgreSQL implementation with UPSERT support

### Interfaces
- `internal/interfaces/http/rewrite_middleware.go` — `ResolverMiddleware`: looks up path in rewrite repo, injects URLRewrite into context
- `internal/interfaces/http/rewrite_handler.go` — `RewriteHandler.Resolve()`: catch-all handler that returns resolved entity type + ID

### Tests
- `internal/domain/routing/routing_test.go` — 7 tests: constructor validation (4), FromDB reconstruction (1), context round-trip (2)
- `internal/interfaces/http/rewrite_test.go` — 5 tests: middleware match/no-match/error (3), handler resolve/missing (2)

## Design Decisions
- Middleware passes through to next handler when no rewrite is found (lets existing routes work normally)
- Catch-all handler returns entity type + ID as JSON; upstream consumers (API composers, frontend) decide how to render
- Context pattern follows existing `auth.WithIdentity`/`auth.IdentityFrom` convention
- UPSERT on Save allows both insert and update of rewrites via single method
