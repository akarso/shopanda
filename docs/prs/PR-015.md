# PR-015 — Variant Domain + Repository

## Summary

Adds the Variant entity, repository interface, event definitions, PostgreSQL implementation, migration, and tests. Variants are sellable units of a product (e.g. size M, color red).

## Design Decisions

- **Separate repository interface**: `VariantRepository` in its own file, parallel to `ProductRepository`. No pagination on `ListByProductID` — a product has a bounded number of variants.
- **`ListByProductID` ordered by `created_at ASC`**: Variants are shown in insertion order (oldest first), unlike products which list newest first.
- **CASCADE delete**: Variant rows are deleted when the parent product is deleted.
- **SKU uniqueness**: Global unique constraint on `sku` — enforced at DB level, detected via `pq.Error` SQLSTATE 23505 → `apperror.Conflict`.
- **Name field**: Optional (`DEFAULT ''`), allows naming variants like "Size M / Red" for display.
- **Same scan pattern**: Reuses the `scanner` interface from `product_repo.go` for `*sql.Row` / `*sql.Rows` polymorphism.

## Migration Schema

```sql
CREATE TABLE variants (
    id          UUID PRIMARY KEY,
    product_id  UUID NOT NULL REFERENCES products(id) ON DELETE CASCADE,
    sku         TEXT UNIQUE NOT NULL,
    name        TEXT NOT NULL DEFAULT '',
    attributes  JSONB NOT NULL DEFAULT '{}'::jsonb,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_variants_product_id ON variants (product_id);
```
