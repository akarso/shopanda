# PR-102 — Roles + Permissions Model

## Phase
29 — Admin RBAC

## Goal
Introduce fine-grained admin roles (admin, manager, editor, support) with a
static permission model. Each role maps to a fixed set of permission strings.
This PR provides the domain layer; PR-103 adds the `RequirePermission()`
middleware and plugin integration.

## Design Decisions

- **Static mapping**: Role-permission mapping is defined in code, not in the DB. The spec forbids dynamic roles (ROLES.md §7). This avoids a join table and keeps permission checks allocation-free.
- **No new DB table**: Roles remain a TEXT column on the `customers` table with a CHECK constraint. The migration simply widens the allowed values.
- **Identity-level roles**: New roles are added to `identity.Role` so that the existing JWT/token pipeline, `RequireRole()`, and `IdentityFrom()` work without changes.
- **Separate `rbac` package**: Keeps permission logic in the domain layer, decoupled from HTTP middleware (added in PR-103).
