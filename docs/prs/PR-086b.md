# PR-086b — Cart Optimistic Locking

## Summary

Adds optimistic concurrency control to the cart repository so that two
concurrent load→modify→save cycles on the same cart no longer silently
overwrite each other. The second writer receives a `409 Conflict` error.

## Background

All five cart mutation methods (`AddItem`, `UpdateItemQuantity`, `RemoveItem`,
`ApplyCoupon`, `RemoveCoupon`) follow a load→modify→save pattern with no
version guard. Two concurrent requests on the same cart could produce a
last-writer-wins race. `payment_repo` and `shipping_repo` already use
`WHERE updated_at = $old` checks — this PR brings the cart repo in line using a
dedicated `version` column.

## Design

### Version column strategy

A dedicated integer counter was chosen over `updated_at`-based locking because:

1. The cart `Save` uses `ON CONFLICT (id) DO UPDATE` (upsert), not a plain
   `UPDATE`. With an upsert, `WHERE updated_at = $old` in the conflict clause
   silently turns the update into a no-op if the timestamp doesn't match — the
   row already exists so no INSERT happens and the WHERE blocks the UPDATE. A
   version counter behaves identically but the semantics are clearer.
2. `updated_at` is set by domain methods — multiple mutations in the same
   millisecond could collide. A monotonic counter avoids this entirely.

### SQL mechanics

```sql
INSERT INTO carts (id, …, version, …)
VALUES ($1, …, $6, …)
ON CONFLICT (id) DO UPDATE SET
    …,
    version = carts.version + 1,
    …
WHERE carts.version = EXCLUDED.version
RETURNING version
```

- **INSERT (new cart):** succeeds, `version` starts at `1` (from `NewCart` and
  DB default). `RETURNING` gives back `1`.
- **UPDATE (existing cart):** succeeds only if the DB `version` matches the
  value the caller loaded (`EXCLUDED.version`). On success, `version` is
  atomically incremented. `RETURNING` gives back the new value.
- **Conflict:** `sql.ErrNoRows` (no row returned) → return
  `apperror.Conflict("cart was modified concurrently")`.
- **Post-commit:** `c.Version` is set to the returned value so the in-memory
  aggregate stays in sync with the database.

### Error propagation

`apperror.Conflict` already maps to HTTP `409 Conflict` via the standard error
middleware. No handler changes are needed — clients can retry.

### Domain field

`Cart.Version` is a public int field, initialised to `1` by `NewCart` and
populated by the repository on read. Domain methods do not touch it — the repo
handles the increment atomically in SQL.

### Migration

```sql
ALTER TABLE carts ADD COLUMN version INT NOT NULL DEFAULT 1;
```

Existing rows get `version = 1`. The first save after migration will match
`version = 1` and bump to `2`.
