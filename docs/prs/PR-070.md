# PR-070 — Attribute Model

## Summary

Introduces a typed attribute system with `Attribute`, `AttributeGroup`, and
`AttributeRegistry` in the catalog domain. Products can be assigned attribute
groups, and each group defines which attributes apply. The registry provides
typed validation of attribute values against their declarations.

## Design

### Attribute

```text
Code     string        — unique identifier (e.g. "color", "weight")
Label    string        — human-readable label
Type     AttributeType — text | number | boolean | select
Required bool          — whether a value must be provided
Options  []string      — allowed values for select type
```

`Validate(value)` checks: nil handling (required vs optional), type correctness,
and option constraint for select attributes.

### AttributeGroup

Groups a set of attribute codes. A product can be assigned one or more groups.
Each group defines which attributes are expected for that product.

### AttributeRegistry

Thread-safe (sync.RWMutex) in-memory registry following the AdminRegistry pattern:

- `RegisterAttribute(attr)` — register/replace by code
- `RegisterGroup(group)` — validates all attribute codes exist before registering
- `GroupAttributes(code)` — resolves group → full Attribute list
- `ValidateAttributes(groupCode, values)` — validates a map of values against group schema
- All retrievals return deep copies to prevent external mutation

## Relationship to Existing Code

Products and variants currently store attributes as flat `map[string]interface{}`
in JSONB columns. This PR introduces the **schema layer** — the registry defines
what attributes exist and what types they expect. A future PR can wire validation
into product create/update flows by checking attribute values against assigned groups.
