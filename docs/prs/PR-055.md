# PR-055 — Search HTTP Endpoint

## Summary

Adds the public `GET /api/v1/search` endpoint that exposes the `search.SearchEngine` (PR-054) over HTTP.

## Changes

### HTTP Handler

- `internal/interfaces/http/search.go` — `SearchHandler`
  - `Search()` → `GET /api/v1/search`
  - Query parameters:
    - `q` — full-text search term (optional)
    - `limit` — results per page (optional, domain handles default/clamp)
    - `offset` — pagination offset (optional)
    - `sort` — sort field: `name`, `-name`, `created_at`, `-created_at` (optional, defaults to relevance when `q` is set)
    - `category` — category filter (optional)
  - Returns `{ products, total, facets }` in standard response envelope
  - Validates `limit`/`offset` as non-negative integers before passing to engine
  - Delegates all search logic to `search.SearchEngine`

### Wiring

- `cmd/api/main.go`

## Design Decisions

- **Public endpoint**: Search is read-only and doesn't require authentication, matching the WEB_API.md spec (`GET /search`).
- **Query params over POST body**: Follows REST conventions for idempotent read operations. Query parameters are cache-friendly and linkable.
- **Handler validates HTTP concerns only**: `limit`/`offset` are parsed and validated as integers. All domain validation (clamping, defaults) is delegated to `SearchQuery.Validate()` / `EffectiveLimit()`.
- **Category filter via query param**: `?category=footwear` maps to `Filters["category"]`. Price filter support deferred (requires structured input better suited to POST or nested params).

## Dependencies

- PR-053 (search domain) — defines `SearchQuery`, `SearchResult`, `SearchEngine` interface
