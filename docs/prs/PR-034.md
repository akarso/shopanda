# PR-034: Order Domain + Storage

## Summary

Introduces the order domain — the immutable purchase snapshot created during checkout. Defines the `Order` aggregate with status lifecycle (pending → confirmed → paid / cancelled / failed), `Item` value objects with variant snapshots (SKU, name, price captured at order time), domain events, repository interface, and a Postgres implementation with migration. Wired into `main.go` for use by the upcoming checkout workflow.

## Changes

### Domain Layer

- **`internal/domain/order/order.go`** *(new)* — `Order` aggregate root with `ID`, `CustomerID`, `Currency`, `Items`, `TotalAmount` (computed from items). Status lifecycle: `Confirm()`, `MarkPaid()`, `Cancel()`, `Fail()`. `SetStatusFromDB()` for hydration. `OrderStatus` value object with `IsValid()`.
- **`internal/domain/order/item.go`** *(new)* — `Item` value object capturing variant snapshot: `VariantID`, `SKU`, `Name`, `Quantity`, `UnitPrice`. `NewItem()` validates all fields. `LineTotal()` returns `UnitPrice * Quantity` with overflow checking.
- **`internal/domain/order/events.go`** *(new)* — Event constants (`order.created`, `order.confirmed`, `order.paid`, `order.cancelled`, `order.failed`) + payload structs (`OrderCreatedData`, `OrderStatusChangedData`).
- **`internal/domain/order/repository.go`** *(new)* — `OrderRepository` interface: `FindByID`, `FindByCustomerID`, `Save`, `UpdateStatus`.

### Infrastructure Layer

- **`migrations/010_create_orders.sql`** *(new)* — `orders` table (UUID PK, customer_id NOT NULL, status with CHECK constraint, BIGINT total, TIMESTAMPTZ timestamps) + `order_items` table (composite PK on order_id + variant_id, SKU/name snapshots, BIGINT price). Indexes on `customer_id` and `status`.
- **`internal/infrastructure/postgres/order_repo.go`** *(new)* — Postgres implementation of `OrderRepository`. Transactional `Save` (insert order + items atomically). `UpdateStatus` updates only status + updated_at with rows-affected check. `FindByID` and `FindByCustomerID` with item loading.

### Wiring

- **`cmd/api/main.go`** — `orderRepo` instantiated and ready for checkout service (PR-035+).

## Design Decisions

- **Orders are immutable snapshots**: Once created, item data (SKU, name, price, quantity) is never recomputed. Only the order status transitions via `UpdateStatus`.
- **Status lifecycle enforced in domain**: `Confirm()` requires `pending`, `MarkPaid()` requires `confirmed`, `Cancel()` allows `pending` or `confirmed`, `Fail()` requires `pending`. Invalid transitions return errors.
- **No HTTP endpoints yet**: Per the spec, orders are created exclusively through the checkout workflow (PR-035+). Direct order creation via API is not permitted.
- **Variant snapshots in items**: `SKU` and `Name` are captured at order time so the order remains valid even if the variant is later modified or deleted.
- **Money stored as BIGINT**: Follows the established pattern (carts, prices) — smallest currency unit (cents) stored as `BIGINT`, currency code as `TEXT`.
- **Composite PK on order_items**: `(order_id, variant_id)` prevents duplicate variants per order, matching the cart_items pattern.
