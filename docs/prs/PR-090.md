# PR-090 — Wire Entity Slugs

## Phase
Phase 23 — URL Routing & CMS

## Scope
Product/category slugs registered in rewrite table on create/update.

## Files

### Domain
- `internal/domain/catalog/events.go` — Added `EventCategoryCreated`, `EventCategoryUpdated` constants and `CategoryCreatedData`, `CategoryUpdatedData` payload structs

### Application
- `internal/application/rewrite/subscriber.go` — `Subscriber` listens to product/category create/update events and saves URL rewrites via `routing.RewriteRepository`

### Wiring
- `cmd/api/main.go` — Instantiates `RewriteRepo`, creates `rewrite.Subscriber`, calls `Register(bus)` to wire all 4 event handlers

### Tests
- `internal/application/rewrite/subscriber_test.go` — 7 tests: product created/updated (2), category created/updated (2), wrong data type (1), repo error (1), Register wiring (1)

## Design Decisions
- Handlers are registered as **sync** (`bus.On`) so a rewrite save failure propagates to the caller (e.g., product admin handler) — the slug path is a critical part of entity creation
- Slug paths are prefixed with `/` (e.g., slug `nike-air-max` → path `/nike-air-max`) to match URL resolution in `ResolverMiddleware`
- UPSERT semantics in `RewriteRepo.Save` means slug changes on update automatically replace the old rewrite
- Category events are defined but will only fire once category admin CRUD or importer publishes them — the subscriber is ready
