# PR-100 — Cache Headers + CDN Config

## Phase
28 — Performance & CDN

## Goal
Add `Cache-Control` response header middleware that differentiates cacheable
(public GET) from non-cacheable (stateful/authenticated) routes, and introduce
`cdn.base_url` configuration for future asset delivery.

## Changes

| File | Change |
|------|--------|
| `internal/platform/config/config.go` | Added `CDNConfig` struct with `BaseURL` field; added `CDN CDNConfig` to `Config`; wired defaults, env overlay (`SHOPANDA_CDN_BASE_URL`), and `flatten()` entry (`cdn.base_url`) |
| `internal/interfaces/http/cache_middleware.go` | New `CacheControlMiddleware(noCachePrefixes)` — sets `public, max-age=300` for cacheable GETs, `no-store` for write methods and non-cacheable paths |
| `internal/interfaces/http/cache_middleware_test.go` | 6 test cases: cacheable GET, no-cache prefix GET, POST no-store, HEAD cacheable, PUT/DELETE/PATCH no-store, storefront SSR cacheable |
| `cmd/api/main.go` | Wired `CacheControlMiddleware` with no-cache prefixes: carts, checkout, orders, account, auth, shipping, admin |

## Design Decisions

- **Global middleware with prefix-based exclusion**: simpler than per-route wrappers; no-cache prefixes are explicit and easy to audit.
- **300s max-age**: matches the spec in `PERFORMANCE.md` — 5 minutes for product/category/page responses.
- **HEAD treated like GET**: HEAD requests hit the same cacheable resources.
- **CDN config is additive**: `cdn.base_url` is optional and defaults to empty. When present, it can be used by media/asset handlers to rewrite URLs. No functional change when absent.
