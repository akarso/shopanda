## Design

### Promotion entity

```go
type Promotion struct {
    ID          string
    Name        string
    Type        PromotionType   // "catalog" or "cart"
    Priority    int             // lower = higher priority
    Active      bool
    StartAt     *time.Time      // nil = immediately active
    EndAt       *time.Time      // nil = no expiry
    Conditions  []byte          // JSON-encoded condition config
    Actions     []byte          // JSON-encoded action config
    CouponBound bool            // requires coupon code to activate
    CreatedAt   time.Time
    UpdatedAt   time.Time
}
```

- `Conditions`/`Actions` are opaque `[]byte` (JSON). The application layer
  (PR-086) interprets them via the rule system from PR-084.
- `IsEligible(now)` checks `Active` + time window.

### Coupon entity

```go
type Coupon struct {
    ID          string
    Code        string          // uppercase alphanumeric + hyphens, 3-50 chars
    PromotionID string
    UsageLimit  int             // 0 = unlimited
    UsageCount  int
    Active      bool
    CreatedAt   time.Time
    UpdatedAt   time.Time
}
```

- `CanRedeem()` checks `Active` + usage limit.
- `Redeem()` increments `UsageCount`, errors if not redeemable.
- Code format validated by regex: `^[A-Z0-9][A-Z0-9\-]{1,48}[A-Z0-9]$`

### Migration

- `promotions` table with JSONB `conditions`/`actions`, composite index on
  `(type, active)`.
- `coupons` table with FK to `promotions`, unique constraint on `code`.

### Repository interfaces

- `PromotionRepository`: `FindByID`, `ListActive(type)`, `Save`, `Delete`
- `CouponRepository`: `FindByCode`, `FindByID`, `ListByPromotion`, `Save`, `Delete`
