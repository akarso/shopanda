# PR-304 — Storefront Cart Structured Error Logging

## Summary

Added structured error logging for storefront cart 500 paths so failures on routes like `/cart/add` produce a JSON error event in the same stdout stream as the existing `http.request` access log.

This closes the debugging gap where `./shopanda serve` showed the request summary with status `500`, but ordinary non-panic cart failures did not emit a matching structured error event with the underlying cause.

## Scope

### Storefront cart error logging

- updated `internal/interfaces/http/storefront_cart.go`
- added structured error events for internal cart failures in cart page rendering, add-to-cart, update-cart, remove-cart, and HTMX cart response rendering
- attached request context to those error events, including method, path, and `request_id` when present
- kept validation and not-found responses unchanged so only internal server failures emit the new error log lines

### Storefront logger injection

- updated `internal/interfaces/http/storefront.go`
- added `WithLog(...)` so tests and custom wiring can override the default storefront logger cleanly

### Regression coverage

- updated `internal/interfaces/http/storefront_cart_test.go`
- added a focused test that triggers an internal `/cart/add` failure, captures the emitted JSON log line, and verifies the error event name, `request_id`, request context, and underlying error message

## Files Changed

- `internal/interfaces/http/storefront.go`
  - added `WithLog(...)` to override the storefront logger for tests or explicit wiring
- `internal/interfaces/http/storefront_cart.go`
  - logged structured error events for internal cart failures before returning HTTP 500
- `internal/interfaces/http/storefront_cart_test.go`
  - added regression coverage proving `/cart/add` emits a structured error log entry for internal failures

## Validation

- `go test ./internal/interfaces/http -run TestStorefrontHandler_CartAdd_LogsInternalError -count=1`

## Notes

- when running `./shopanda serve`, these structured cart error logs are written to the same stdout stream as the existing JSON request logs
- this PR is intentionally narrow: it fixes the missing structured logging in the storefront cart flow without widening into a broader logging pass across all storefront handlers
