# PR-302 — Deployment Service Manager Guidance

## Summary

Expanded the deployment guide so operators can run the Shopanda binary as a managed host service without guessing about file layout, privileges, environment handling, or day-2 operations.

This follow-up closes the gaps in the bare-metal section by documenting:

- where the binary, config file, and service env file should live
- how to keep secrets out of the repository checkout
- which file is safe for non-secret config versus which file is intended for sensitive values
- which user should own and run the processes
- quick background commands for temporary bring-up
- concrete `systemd` management commands
- a short FreeBSD `rc.d` example alongside the Linux service examples

## Scope

### Bare-metal deployment documentation

- updated `docs/guides/DEPLOYMENT.md`
- documented a dedicated non-root `shopanda` service account
- added an example host layout under `/opt/shopanda`, `/etc/shopanda`, and `/var/log/shopanda`
- documented that the binary still expects a real `configs/config.yaml` in its working directory
- documented copying `.env.example` or an installer-generated `.env` into `/etc/shopanda/shopanda.env`
- documented removing the repo-root production `.env` after copying it into the service location
- documented the permission boundary between world-readable `config.yaml` and `0640`-protected `shopanda.env`
- documented that secrets belong only in `shopanda.env` or another protected secret store, not in `config.yaml`

### Service operation guidance

- added explicit `sudoedit` commands for editing `config.yaml` and `shopanda.env`
- added setup, migrate, and seed examples that run under the service account with the service env loaded
- added `nohup`-based background commands for `serve`, `worker`, and `scheduler`
- added `systemctl` and `journalctl` commands for enable, start, restart, status, and log following

### FreeBSD example

- added a short `rc.d` example for `shopanda_web`
- documented the matching `rc.conf` flags
- noted how to duplicate the same pattern for `shopanda_worker` and `shopanda_scheduler`
- clarified why the FreeBSD example uses `/usr/local/etc/rc.d/shopanda_web` and `/usr/local/etc/shopanda.env` while the Linux examples use `/etc/shopanda/shopanda.env`

## Files Changed

- `docs/guides/DEPLOYMENT.md`
  - expanded the bare-metal and service-manager guidance with concrete installation, env-file, secret-boundary, background-run, `systemd`, and FreeBSD `rc.d` instructions

## Notes

- this PR is documentation-only and does not change runtime behavior
- the guide now aligns more closely with the current config loader behavior by explicitly documenting the need for a real `config.yaml` in the working directory, treating repo-root `.env` as a development convenience rather than a production service pattern, and separating non-secret config from protected service secrets
