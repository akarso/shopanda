# PR-302 — Deployment Service Manager Guidance

## Summary

Expanded the deployment guide so operators can run the Shopanda binary as a managed host service without guessing about file layout, privileges, environment handling, or day-2 operations.

This follow-up closes the gaps in the bare-metal section by documenting:

- where the working directory, config file, and service env file should live
- how to keep secrets out of the repository checkout
- which file is safe for non-secret config versus which file is intended for sensitive values
- which user should own and run the processes
- quick background commands for temporary bring-up
- concrete `systemd` management commands
- a short FreeBSD `rc.d` example alongside the Linux service examples
- that the simplest host deployment keeps the binary and runtime assets in the existing working directory
- the shell-export detail needed for manual `sh -c` examples that source the service env file

It also aligns the shipped configuration example with that guidance by removing secret-bearing examples from `config.example.yaml` and pointing operators to environment variables for sensitive values.

## Scope

### Bare-metal deployment documentation

- updated `docs/guides/DEPLOYMENT.md`
- documented a dedicated non-root `shopanda` service account
- documented the simpler working-directory-first deployment pattern using the existing checkout or deploy directory
- documented that the binary still expects a real `configs/config.yaml` in its working directory
- documented copying `.env.example` or an installer-generated `.env` into `/etc/shopanda/shopanda.env`
- documented removing the repo-root production `.env` after copying it into the external secret location
- documented the boundary between non-secret `configs/config.yaml` and `0640`-protected `shopanda.env`
- documented that secrets belong only in `shopanda.env` or another protected secret store, not in `config.yaml`
- documented that service env files must remain plain `KEY=value` entries with no spaces around `=` and should quote shell-sensitive values
- documented that `serve`, `worker`, and `scheduler` normally share the same `shopanda.env` file unless intentionally overridden

### Configuration example cleanup

- updated `configs/config.example.yaml`
- added a top-level note that secrets should stay out of YAML
- removed inline example values for secret-bearing fields such as database password, SMTP password, S3 credentials, and Meilisearch API key
- clarified that webhook secrets and similar credentials should be supplied via environment variables

### Service operation guidance

- added explicit `sudoedit` commands for editing `config.yaml` and `shopanda.env`
- added setup, migrate, and seed examples that run under the service account with the service env loaded
- added `nohup`-based background commands for `serve`, `worker`, and `scheduler`
- added `systemctl` and `journalctl` commands for enable, start, restart, status, and log following
- corrected the manual shell examples to use `set -a` so sourced env-file values are exported to the Shopanda process
- updated the examples to run directly from the working directory instead of assuming a copied `/opt` install tree

### FreeBSD example

- added a short `rc.d` example for `shopanda_web`
- documented the matching `rc.conf` flags
- noted how to duplicate the same pattern for `shopanda_worker` and `shopanda_scheduler`
- clarified why the FreeBSD example uses `/usr/local/etc/rc.d/shopanda_web` and `/usr/local/etc/shopanda.env` while the Linux examples use `/etc/shopanda/shopanda.env`
- updated the FreeBSD example to use the same `set -a` export pattern as the Linux shell examples

## Files Changed

- `docs/guides/DEPLOYMENT.md`
  - expanded the bare-metal and service-manager guidance with a working-directory-first deployment flow, external secret-file handling, background-run examples, `systemd`, and FreeBSD `rc.d` instructions
- `configs/config.example.yaml`
  - removed secret-bearing example values and clarified that sensitive settings belong in environment variables or another protected secret store

## Notes

- this PR is documentation-only and does not change runtime behavior
- the guide now aligns more closely with the current runtime behavior by explicitly documenting the need for a real `config.yaml` in the working directory, the difference between development `.env` usage and production service env files, and the separation between non-secret config and protected service secrets
