# PR-303 — Payments Migration Order UUID Fix

## Summary

Fixed a broken PostgreSQL migration that declared `payments.order_id` as `TEXT` even though `orders.id` is `UUID`.

This mismatch caused `./shopanda migrate` to stop at `012_create_payments.sql` with PostgreSQL error `42804`, which in turn prevented later schema changes from being applied. In production, that left the database missing tables created by later migrations, including `stores`, and surfaced as storefront failures like `pq: relation "stores" does not exist`.

## Scope

### Payments schema correction

- updated `migrations/012_create_payments.sql`
- changed `payments.order_id` from `TEXT` to `UUID`
- kept the foreign key to `orders(id)` intact, but aligned its column type with the referenced primary key
- restored migration consistency with the order schema introduced in `010_create_orders.sql`

### Integration test alignment

- updated `internal/infrastructure/postgres/payment_repo_test.go`
- stopped using fake non-UUID order IDs in payment repository integration tests
- inserted a real order row before creating payments so the tests exercise the actual foreign-key contract
- cleaned up inserted `orders` rows alongside `payments` rows in the touched tests

## Files Changed

- `migrations/012_create_payments.sql`
  - corrected `payments.order_id` to use `UUID` so the foreign key to `orders(id)` can be created successfully
- `internal/infrastructure/postgres/payment_repo_test.go`
  - updated the integration tests to create valid parent orders and use real UUID order IDs

## Validation

- `go test ./internal/infrastructure/postgres -run TestPaymentRepo -count=1`

## Notes

- this PR fixes the immediate migration blocker that prevented the schema from progressing past payments
- the change is intentionally narrow: it corrects the broken payments migration and the nearby tests without widening into broader order-id cleanup in unrelated modules
