# PR-303 — Migration Foreign Key UUID Fixes

## Summary

Fixed broken PostgreSQL migrations that declared foreign-key columns as `TEXT` even though the referenced primary keys are `UUID`.

These mismatches caused `./shopanda migrate` to stop with PostgreSQL error `42804`, which in turn prevented later schema changes from being applied. In production, that left the database missing tables created by later migrations, including `stores`, and surfaced as storefront failures like `pq: relation "stores" does not exist`.

## Scope

### Payments schema correction

- updated `migrations/012_create_payments.sql`
- changed `payments.order_id` from `TEXT` to `UUID`
- kept the foreign key to `orders(id)` intact, but aligned its column type with the referenced primary key
- restored migration consistency with the order schema introduced in `010_create_orders.sql`

### Category join-table schema correction

- updated `migrations/014_create_categories.sql`
- changed `product_categories.product_id` from `TEXT` to `UUID`
- kept the foreign key to `products(id)` intact, but aligned its column type with the referenced primary key
- restored migration consistency with the product schema introduced in `001_create_products.sql`

### Integration test alignment

- updated `internal/infrastructure/postgres/payment_repo_test.go`
- stopped using fake non-UUID order IDs in payment repository integration tests
- inserted a real order row before creating payments so the tests exercise the actual foreign-key contract
- cleaned up inserted `orders` rows alongside `payments` rows in the touched tests

## Files Changed

- `migrations/012_create_payments.sql`
  - corrected `payments.order_id` to use `UUID` so the foreign key to `orders(id)` can be created successfully
- `migrations/014_create_categories.sql`
  - corrected `product_categories.product_id` to use `UUID` so the foreign key to `products(id)` can be created successfully
- `internal/infrastructure/postgres/payment_repo_test.go`
  - updated the integration tests to create valid parent orders and use real UUID order IDs

## Notes

- this PR fixes the immediate migration blockers that prevented the schema from progressing through payments and category joins
- the change is intentionally narrow: it corrects the broken foreign-key column types and the nearby payment repo tests without widening into broader identifier cleanup in unrelated modules
