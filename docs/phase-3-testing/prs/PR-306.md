# PR-306 — Guest Cart Claim And Merge On Login/Register

## Summary

Fixed the storefront split where an anonymous shopper could add items to a guest cart, then log in or register and immediately see an empty authenticated cart while other storefront surfaces still reflected the guest cart state.

This PR adds a dedicated cart handoff use case and wires it into storefront login and registration so the anonymous cart is claimed by the customer when no active customer cart exists, or merged into the active customer cart when one already exists. The guest cart cookie is then cleared so the storefront no longer has two competing cart identities for the same browser session.

## Scope

### Cart handoff use case

- updated `internal/application/cart/service.go`
- added `ClaimGuestCart(ctx, guestCartID, customerID)`
- when the customer has no active cart, the guest cart is reassigned to that customer
- when the customer already has an active cart in the same currency, guest items are merged into the active cart and the guest cart is deleted
- preserved applied guest coupon state when the target customer cart does not already carry a coupon
- treated stale or foreign-owned cart cookies as no-op inputs instead of trying to claim another customer's cart

### Storefront auth integration

- updated `internal/interfaces/http/storefront_account.go`
- login and registration now call the cart handoff before setting the storefront session cookie
- if a real cart sync failure occurs, the auth response stops and returns an error page instead of silently leaving cart state split
- added guest cart cookie clearing after successful claim or merge so `/cart`, minicart, and later requests resolve against a single cart identity

### Cookie helpers

- updated `internal/interfaces/http/storefront_cart.go`
- added a dedicated helper for clearing the guest cart cookie alongside the existing cookie setter

### Regression coverage

- updated `internal/application/cart/service_test.go`
- added focused service coverage for:
  - guest cart claimed into an empty customer cart state
  - guest cart merged into an existing customer cart
- updated `internal/interfaces/http/storefront_account_test.go`
- added storefront coverage proving both login and registration claim the guest cart and make it visible through the authenticated cart flow

## Notes

- this PR intentionally does not implement guest checkout; checkout still requires an authenticated customer and that larger refactor is tracked separately in `docs/phase-3-testing/ROADMAP.md`
- the chosen behavior follows the common Magento-style continuity path: guest cart first, customer cart after auth, one resulting active cart after handoff
- clearing the guest cart cookie after a successful handoff is necessary to avoid leaking the now-customer-owned cart back into a later anonymous browser state such as post-logout browsing
