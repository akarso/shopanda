# PR-209 — Stripe refunds

## Goal

Implement Stripe refund initiation via an admin endpoint and
`charge.refunded` webhook reconciliation, completing the refund
lifecycle defined in STRIPE_PAYMENT.md §4.

## Scope

| Area | Change |
|---|---|
| `internal/domain/payment/refunder.go` | New `Refunder` interface (`Refund(ctx, providerRef, amount, currency) (RefundResult, error)`) and `RefundResult` struct — kept separate from `Provider` to avoid forcing all providers to implement refunds |
| `internal/infrastructure/stripepay/provider.go` | Added `Refund()` method — POSTs form-encoded request to `/v1/refunds` with `payment_intent`, `amount`, `Idempotency-Key: refund:{providerRef}:{amount}`, validates response `id` field |
| `internal/interfaces/http/refund.go` | New `RefundHandler` — `POST /api/v1/admin/orders/{orderId}/refund`; validates body, checks amount ≤ payment, calls refunder, transitions status, publishes event |
| `internal/interfaces/http/stripe_webhook.go` | Added `PaymentIntent` field to `stripeEvent.Data.Object`, `charge.refunded` routing, `handleChargeRefunded()` — looks up payment via `metadata.payment_id`, idempotent for already-refunded |
| `cmd/api/main.go` | Wires `RefundHandler` conditionally via `payProvider.(payment.Refunder)` type assertion; adds `requireOrdersWrite` permission middleware; registers admin refund route |

## Design notes

1. **Separate Refunder interface** — `Refunder` is not embedded in
   `Provider` because not all payment providers support programmatic
   refunds (e.g. manual payments). The `cmd/api/main.go` wiring uses a
   type assertion `payProvider.(payment.Refunder)` to conditionally
   enable the refund endpoint only when the active provider supports it.

2. **Idempotency key** — Stripe refund requests use
   `Idempotency-Key: refund:{providerRef}:{amount}` to prevent duplicate
   refunds for the same PaymentIntent and amount combination.

3. **Admin endpoint authorization** — The refund route is gated behind
   `rbac.OrdersWrite` permission, consistent with other admin order
   operations.

4. **Webhook reconciliation** — `charge.refunded` events from Stripe are
   handled by looking up the payment via `metadata.payment_id` (propagated
   from the PaymentIntent). If the payment is already refunded (e.g. by
   the admin endpoint), the handler returns `already_processed`.

5. **Amount validation** — The admin handler validates
   `req.Amount > 0` and `req.Amount <= p.Amount.Amount()` to prevent
   zero/negative refunds and over-refunds.

6. **Status transition** — Only `completed → refunded` is allowed by the
   domain `Refund()` method. The handler calls the Stripe API first, then
   transitions the domain status. If the Stripe call succeeds but the
   status transition fails (e.g. concurrent webhook already refunded it),
   a validation error is returned — the Stripe refund is already created
   and will reconcile via the webhook.

## Tests

5 tests added to `stripepay/provider_test.go`:

- `TestProvider_Refund_Success` — correct endpoint, form params, idempotency key; returns `RefundResult` with `ProviderRef`
- `TestProvider_Refund_EmptyProviderRef` — error for empty provider ref
- `TestProvider_Refund_APIError` — error propagation for Stripe API errors
- `TestProvider_Refund_MissingID` — error when response lacks `id` field
- `TestProvider_Refund_NetworkError` — error for unreachable server

8 tests in new `interfaces/http/refund_test.go`:

- `TestRefund_Success` — payment transitions to refunded, returns refund details
- `TestRefund_InvalidBody` — 422 for malformed JSON
- `TestRefund_ZeroAmount` — 422 for zero/negative amount
- `TestRefund_PaymentNotFound` — 404 when no payment exists for order
- `TestRefund_AmountExceedsPayment` — 422 when refund exceeds payment amount
- `TestRefund_NoProviderRef` — 422 when payment has no provider reference
- `TestRefund_RefunderError` — 500 when Stripe returns error
- `TestRefund_NotCompleted` — 422 when payment is not in completed status

5 tests added to `interfaces/http/stripe_webhook_test.go`:

- `TestStripeWebhook_ChargeRefunded` — payment transitions to refunded via webhook
- `TestStripeWebhook_ChargeRefunded_AlreadyRefunded` — 200 with `already_processed`
- `TestStripeWebhook_ChargeRefunded_MissingMetadata` — 422 for missing `payment_id`
- `TestStripeWebhook_ChargeRefunded_ProviderMismatch` — 422 for non-Stripe payment
- `TestStripeWebhook_ChargeRefunded_PaymentNotFound` — 404 for unknown payment
