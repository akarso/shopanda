# PR-208 — Stripe webhook handler

## Goal

Implement a dedicated Stripe webhook handler that verifies `Stripe-Signature`
headers and routes `payment_intent.succeeded` / `payment_intent.payment_failed`
events to update payment status, completing the Stripe payment lifecycle
started in PR-207.

## Scope

| Area | Change |
|---|---|
| `internal/infrastructure/stripepay/webhook.go` | New `VerifySignature` function — parses `Stripe-Signature` header (`t=<ts>,v1=<hmac>`), verifies HMAC-SHA256 of `"{ts}.{payload}"`, enforces 5 min timestamp tolerance |
| `internal/interfaces/http/stripe_webhook.go` | New `StripeWebhookHandler` — dedicated handler for `POST /api/v1/payments/webhook/stripe`; parses Stripe event envelope, routes payment_intent events; idempotency decided by inspecting payment status (not error return) |
| `internal/platform/config/config.go` | Added `WebhookSecret` to `StripeConfig`, env overlay for `SHOPANDA_PAYMENT_STRIPE_WEBHOOK_SECRET` |
| `cmd/api/main.go` | Wires `StripeWebhookHandler` when Stripe enabled + `SHOPANDA_PAYMENT_STRIPE_WEBHOOK_SECRET` env var set (YAML `webhook_secret` ignored with warning); registers route before generic `{provider}` handler; removed PR-208 TODO comment |

## Design notes

1. **Stripe-Signature verification** — Stripe signs webhooks using
   `HMAC-SHA256("{timestamp}.{raw_body}", webhook_secret)`. The header
   format is `t=<unix_ts>,v1=<hex>[,v1=<hex>]`. The verifier checks all
   `v1` signatures (Stripe may include multiple during secret rotation),
   and rejects timestamps older/newer than 5 minutes to prevent replay
   attacks.

2. **Dedicated handler vs adapter** — The `StripeWebhookHandler` is a
   standalone handler rather than an adapter to the generic
   `PaymentWebhookHandler`. Stripe's event envelope (`{id, type, data}`)
   and signature format are fundamentally different from the generic
   `{payment_id, provider_ref, success}` + HMAC envelope. Sharing code
   would add unnecessary coupling. The generic handler remains available
   for other providers.

3. **Idempotent processing** — The handler checks `p.Status() != StatusPending`
   before calling `Complete()`/`Fail()`. If the payment is no longer pending
   (already completed or failed), it returns `200 OK` with
   `{"status": "already_processed"}`. If the payment is still pending but
   `Complete()`/`Fail()` returns an unexpected error, that is treated as a
   real failure and returns 500. This ensures idempotency is decided by
   observed state, not by swallowing domain errors.

4. **Unhandled events return 200** — Stripe sends many event types. The
   handler acknowledges unhandled types with `200 OK {"status": "ignored"}`
   to prevent Stripe from retrying them.

5. **Payment lookup via metadata** — The Stripe PaymentIntent carries
   `metadata.payment_id` (set during `Initiate()` in PR-207). The webhook
   handler extracts this to find the payment record. Missing metadata
   returns a 422.

6. **Route priority** — `POST /api/v1/payments/webhook/stripe` is
   registered before the generic `POST /api/v1/payments/webhook/{provider}`.
   Go 1.22+ net/http routes exact patterns over parameterized ones, so the
   Stripe handler always wins for `stripe` while other providers still hit
   the generic handler.

7. **Webhook secret sourced from env** — Like the Stripe API key (PR-207),
   the webhook secret is sourced exclusively from the
   `SHOPANDA_PAYMENT_STRIPE_WEBHOOK_SECRET` env var. YAML `webhook_secret`
   values are ignored with a `payment.stripe.webhook_secret_ignored` warning.
   When Stripe is enabled but the env var is missing, a warning is logged
   and Stripe webhooks will not be handled.

## Tests

10 tests in `stripepay/webhook_test.go` (all new):

- `TestVerifySignature_Valid` — correct signature accepted
- `TestVerifySignature_MultipleV1` — correct signature among multiple v1 values
- `TestVerifySignature_EmptyHeader` — returns `ErrSignatureMissing`
- `TestVerifySignature_NoTimestamp` — missing `t=` returns `ErrSignatureInvalid`
- `TestVerifySignature_NoV1` — missing `v1=` returns `ErrSignatureInvalid`
- `TestVerifySignature_BadTimestamp` — non-numeric timestamp returns `ErrSignatureInvalid`
- `TestVerifySignature_WrongSignature` — mismatched HMAC returns `ErrSignatureInvalid`
- `TestVerifySignature_ExpiredTimestamp` — 10 min old timestamp rejected
- `TestVerifySignature_FutureTimestamp` — 10 min future timestamp rejected
- `TestVerifySignature_WithinTolerance` — 2 min old timestamp accepted

12 tests in `interfaces/http/stripe_webhook_test.go` (all new):

- `TestStripeWebhook_PaymentIntentSucceeded` — payment transitions to completed, ProviderRef set
- `TestStripeWebhook_PaymentIntentFailed` — payment transitions to failed
- `TestStripeWebhook_MissingSignature` — 401 when no Stripe-Signature header
- `TestStripeWebhook_InvalidSignature` — 401 when signature doesn't match
- `TestStripeWebhook_AlreadyCompleted` — 200 with `already_processed` (idempotent)
- `TestStripeWebhook_AlreadyFailed` — 200 with `already_processed` (idempotent)
- `TestStripeWebhook_UnhandledEventType` — 200 with `ignored` for unknown event types
- `TestStripeWebhook_MissingEventID` — 422 when event has no `id`
- `TestStripeWebhook_MissingPaymentIDInMetadata` — 422 when metadata lacks `payment_id`
- `TestStripeWebhook_PaymentNotFound` — 404 when payment doesn't exist
- `TestStripeWebhook_ProviderMismatch` — 422 when payment method is not `stripe`
- `TestStripeWebhook_InvalidJSON` — 422 on malformed body
