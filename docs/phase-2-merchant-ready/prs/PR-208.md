# PR-208 ‚Äî Stripe webhook handler

## Goal

Implement a dedicated Stripe webhook handler that verifies `Stripe-Signature`
headers and routes `payment_intent.succeeded` / `payment_intent.payment_failed`
events to update payment status, completing the Stripe payment lifecycle
started in PR-207.

## Scope

| Area | Change |
|---|---|
| `internal/infrastructure/stripepay/webhook.go` | New `VerifySignature` function ‚Äî parses `Stripe-Signature` header (`t=<ts>,v1=<hmac>`), verifies HMAC-SHA256 of `"{ts}.{payload}"`, enforces 5 min timestamp tolerance |
| `internal/interfaces/http/stripe_webhook.go` | New `StripeWebhookHandler` ‚Äî dedicated handler for `POST /api/v1/payments/webhook/stripe`; parses Stripe event envelope, routes payment_intent events, idempotent for already-processed payments |
| `internal/platform/config/config.go` | Added `WebhookSecret` to `StripeConfig`, env overlay for `SHOPANDA_PAYMENT_STRIPE_WEBHOOK_SECRET` |
| `cmd/api/main.go` | Wires `StripeWebhookHandler` when Stripe enabled + webhook secret set; registers route before generic `{provider}` handler; removed PR-208 TODO comment |

## Design notes

1. **Stripe-Signature verification** ‚Äî Stripe signs webhooks using
   `HMAC-SHA256("{timestamp}.{raw_body}", webhook_secret)`. The header
   format is `t=<unix_ts>,v1=<hex>[,v1=<hex>]`. The verifier checks all
   `v1` signatures (Stripe may include multiple during secret rotation),
   and rejects timestamps older/newer than 5 minutes to prevent replay
   attacks.

2. **Dedicated handler vs adapter** ‚Äî The `StripeWebhookHandler` is a
   standalone handler rather than an adapter to the generic
   `PaymentWebhookHandler`. Stripe's event envelope (`{id, type, data}`)
   and signature format are fundamentally different from the generic
   `{payment_id, provider_ref, success}` + HMAC envelope. Sharing code
   would add unnecessary coupling. The generic handler remains available
   for other providers.

3. **Idempotent processing** ‚Äî If the payment is already `completed` or
   `failed`, the handler returns `200 OK` with `{"status": "already_processed"}`.
   This leverages the domain model's existing state transition guards
   (`Complete()` / `Fail()` only work on pending payments) rather than
   storing event IDs. Stripe retries on non-2xx responses, so idempotent
   200s prevent duplicate processing.

4. **Unhandled events return 200** ‚Äî Stripe sends many event types. The
   handler acknowledges unhandled types with `200 OK {"status": "ignored"}`
   to prevent Stripe from retrying them.

5. **Payment lookup via metadata** ‚Äî The Stripe PaymentIntent carries
   `metadata.payment_id` (set during `Initiate()` in PR-207). The webhook
   handler extracts this to find the payment record. Missing metadata
   returns a 422.

6. **Route priority** ‚Äî `POST /api/v1/payments/webhook/stripe` is
   registered before the generic `POST /api/v1/payments/webhook/{provider}`.
   Go 1.22+ net/http routes exact patterns over parameterized ones, so the
   Stripe handler always wins for `stripe` while other providers still hit
   the generic handler.

7. **Webhook secret sourced from env** ‚Äî Like the Stripe API key (PR-207),
   the webhook secret is sourced from `SHOPANDA_PAYMENT_STRIPE_WEBHOOK_SECRET`
   env var. When Stripe is enabled but the webhook secret is missing, a
   warning is logged and Stripe webhooks will not be handled (the generic
   handler would reject `stripe` method events with a signature error).

## Tests

10 tests in `stripepay/webhook_test.go` (all new):

- `TestVerifySignature_Valid` ‚Äî correct signature accepted
- `TestVerifySignature_MultipleV1` ‚Äî correct signature among multiple v1 values
- `TestVerifySignature_EmptyHeader` ‚Äî returns `ErrSignatureMissing`
- `TestVerifySignature_NoTimestamp` ‚Äî missing `t=` returns `ErrSignatureInvalid`
- `TestVerifySignature_NoV1` ‚Äî missing `v1=` returns `ErrSignatureInvalid`
- `TestVerifySignature_BadTimestamp` ‚Äî non-numeric timestamp returns `ErrSignatureInvalid`
- `TestVerifySignature_WrongSignature` ‚Äî mismatched HMAC returns `ErrSignatureInvalid`
- `TestVerifySignature_ExpiredTimestamp` ‚Äî 10 min old timestamp rejected
- `TestVerifySignature_FutureTimestamp` ‚Äî 10 min future timestamp rejected
- `TestVerifySignature_WithinTolerance` ‚Äî 2 min old timestamp accepted

12 tests in `interfaces/http/stripe_webhook_test.go` (all new):

- `TestStripeWebhook_PaymentIntentSucceeded` ‚Äî payment transitions to completed, ProviderRef set
- `TestStripeWebhook_PaymentIntentFailed` ‚Äî payment transitions to failed
- `TestStripeWebhook_MissingSignature` ‚Äî 401 when no Stripe-Signature header
- `TestStripeWebhook_InvalidSignature` ‚Äî 401 when signature doesn't match
- `TestStripeWebhook_AlreadyCompleted` ‚Äî 200 with `already_processed` (idempotent)
- `TestStripeWebhook_AlreadyFailed` ‚Äî 200 with `already_processed` (idempotent)
- `TestStripeWebhook_UnhandledEventType` ‚Äî 200 with `ignored` for unknown event types
- `TestStripeWebhook_MissingEventID` ‚Äî 422 when event has no `id`
- `TestStripeWebhook_MissingPaymentIDInMetadata` ‚Äî 422 when metadata lacks `payment_id`
- `TestStripeWebhook_PaymentNotFound` ‚Äî 404 when payment doesn't exist
- `TestStripeWebhook_ProviderMismatch` ‚Äî 422 when payment method is not `stripe`
- `TestStripeWebhook_InvalidJSON` ‚Äî 422 on malformed body
