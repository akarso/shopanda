# PR-220 — Product Management Pages

## Summary

Implemented admin product management pages in embedded SPA:

- Product grid page (`/admin/products`)
- Product create page (`/admin/products/new`)
- Product edit page (`/admin/products/{id}`)
- Schema-driven form rendering from admin schema API
- Variant inline editing panel on product edit page

No build step. Vanilla JS only.

## Scope

### Product Grid

- Fetches grid schema: `GET /api/v1/admin/grids/product.grid`
- Fetches product data: `GET /api/v1/admin/products?page=1&per_page=20&sort=created_at&order=desc`
- Renders schema columns dynamically
- Includes `New Product` action
- Includes row edit link to `/admin/products/{id}`

### Product Form (Schema-Driven)

- Fetches form schema: `GET /api/v1/admin/forms/product.form`
- Renders fields by schema type:
  - `text`
  - `number`
  - `select`
  - `checkbox`
  - `textarea`
- Create flow:
  - `POST /api/v1/admin/products`
- Edit flow:
  - loads product via `GET /api/v1/products/{id}`
  - saves via `PUT /api/v1/admin/products/{id}`

### Variant Inline Editing

- Visible on edit page only
- Lists variants: `GET /api/v1/products/{id}/variants`
- Creates variant inline: `POST /api/v1/admin/products/{id}/variants`
- Updates variant inline row: `PUT /api/v1/admin/products/{id}/variants/{variantId}`

### Auth & Permissions

All admin and schema endpoints require a valid admin session:

| Endpoint | Required scope |
|---|---|
| `GET /api/v1/admin/grids/product.grid` | products:read |
| `GET /api/v1/admin/forms/product.form` | products:read |
| `GET /api/v1/admin/products` | products:read |
| `POST /api/v1/admin/products` | products:write |
| `PUT /api/v1/admin/products/{id}` | products:write |
| `GET /api/v1/products/{id}/variants` | public (no auth) |
| `POST /api/v1/admin/products/{id}/variants` | products:write |
| `PUT /api/v1/admin/products/{id}/variants/{variantId}` | products:write |

**Common failure responses:**

- `401 Unauthorized` — missing or invalid session token; prompt re-login.
- `403 Forbidden` — authenticated but lacks the required permission; display a "not allowed" message distinct from validation errors.

Consumers should handle `401` and `403` distinctly from `4xx` payload/validation errors (e.g. `400`, `422`) to avoid misleading the user with form-level error messaging on auth failures.

## Files Changed

- `internal/interfaces/http/admin/dist/admin.js`
  - dynamic route resolution for `/admin/products/new` and `/admin/products/{id}`
  - product grid renderer (schema + data)

## Notes

- Schema names currently registered in backend are `product.form` and `product.grid`.
- JSON normalization is required because backend structs serialize with Go field names (`ID`, `CreatedAt`, etc.).
- Kept implementation minimal and deterministic, aligned with no-build SPA constraints.
