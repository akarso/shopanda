# PR-221 — Order Management Pages

## Summary

Implemented admin order management pages in embedded SPA:

- Order grid page (`/admin/orders`)
- Order detail page (`/admin/orders/{id}`)
- Status transition action from order detail
- Backend admin status update endpoint with domain transition rules

No build step. Vanilla JS only.

## Scope

### Order Grid

- Fetches orders: `GET /api/v1/admin/orders?offset=0&limit=50`
- Columns rendered:
  - ID
  - customer
  - total
  - status
  - payment status (derived)
  - date
- Includes status filter (`pending`, `confirmed`, `paid`, `cancelled`, `failed`)
- Includes row detail link to `/admin/orders/{id}`

### Order Detail

- Fetches order: `GET /api/v1/admin/orders/{orderId}`
- Displays:
  - order header (id, status, customer, date)
  - line items (name, SKU, qty, unit price, line total)
  - total amount
  - payment status (derived from order status)
  - shipping section placeholder for missing payload fields

### Status Transitions

- New backend endpoint: `PUT /api/v1/admin/orders/{orderId}`
- Request body: `{ "status": "confirmed|paid|cancelled|failed" }`
- Uses domain transition rules via order methods (`Confirm`, `MarkPaid`, `Cancel`, `Fail`)
- Invalid transitions return validation error (HTTP `422`)

Allowed transitions implemented:

- `pending` -> `confirmed`, `failed`, `cancelled`
- `confirmed` -> `paid`, `cancelled`
- `paid` -> no transitions
- `cancelled` -> no transitions
- `failed` -> no transitions

## Auth & Permissions

All admin order endpoints require authenticated admin access and appropriate order permissions:

- `GET /api/v1/admin/orders` -> orders read
- `GET /api/v1/admin/orders/{orderId}` -> orders read
- `PUT /api/v1/admin/orders/{orderId}` -> orders write

Common failures:

- `401 Unauthorized` for missing/invalid authentication
- `403 Forbidden` for authenticated users without required role/permission
