# PR-204 — File-Based Template Loader

## Summary

Introduces a file-based email template system that loads HTML templates from
`templates/emails/`, supports a shared layout wrapper (`_layout.html`), and
integrates with the existing `mail.Templates` registry.

## Files Changed

### Domain — `internal/domain/mail/`

| File | Change |
|------|--------|
| `mail.go` | Added `ErrTemplateNotFound` sentinel error |
| `loader.go` | **New** — `TemplateLoader` interface, `EmailData` struct |

### Infrastructure — `internal/infrastructure/email/`

| File | Change |
|------|--------|
| `file_loader.go` | **New** — `FileLoader` reads `{dir}/{name}.html`, extracts `<!-- Subject: ... -->` from first line |
| `layout.go` | **New** — `LayoutRenderer` parses `_layout.html` and wraps rendered bodies with store branding |

| File | Change |

## Architecture

```text
templates/emails/
  _layout.html            ← shared wrapper
  order_confirmed.html    ← (added in PR-205)
  ...

internal/domain/mail/
  loader.go               ← TemplateLoader interface + EmailData
  mail.go                 ← ErrTemplateNotFound

internal/infrastructure/email/
  file_loader.go          ← FileLoader (implements TemplateLoader)
  layout.go               ← LayoutRenderer (_layout.html → wrapped body)
  email.go                ← LoadDir, RenderEmail, Setup
```

### Flow

```text
1. Setup(dir) → FileLoader + LayoutRenderer
2. LoadDir(registry, loader, dir) → reads *.html, calls registry.Register()
3. RenderEmail(registry, lr, name, to, data) → Render + Wrap
```

### Resolution Order (design)

```text
1. DB template (future — not in this PR)
2. File template (templates/emails/{name}.html)
3. Fallback: built-in hardcoded (RegisterTemplates in notification service)
```

File templates registered via `LoadDir` silently overwrite hardcoded templates
already in the registry, achieving file → hardcoded fallback ordering.

## Template File Format

```html
<!-- Subject: Order {{.Data.OrderID}} — Confirmation -->
<h1>Thank you, {{.Data.FirstName}}!</h1>
<p>Your order has been confirmed.</p>
```

- First line: `<!-- Subject: ... -->` (required, supports Go template expressions)
- Remaining lines: body HTML (inner content before layout wrapping)

## Key Design Decisions

1. **Subject in HTML comment** — keeps templates as single files, parseable by
   editors and email preview tools.
2. **LayoutRenderer is optional** — when `_layout.html` is absent, `Wrap()`
   returns the body unchanged, enabling layout-free testing and plain emails.
3. **LoadDir skips `_` prefix** — `_layout.html` and future partials are not
   registered as standalone templates.
4. **Domain stays clean** — `TemplateLoader` is a domain interface; file I/O
   lives entirely in `internal/infrastructure/email/`.
