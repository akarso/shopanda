# PR-228 — Checkout Flow

## Summary

Implemented the storefront checkout slice as a server-rendered multi-step flow:

- added `/checkout/address`, `/checkout/shipping`, `/checkout/payment`, and `/checkout/confirm`
- kept the flow SSR-first with normal HTML forms and no storefront build step
- reused the existing checkout application service to place orders at the final step
- rendered provider-aware shipping and payment views from the configured runtime providers

## Scope

### Storefront Checkout Steps

- added an address step with required field validation
- added a shipping step that renders the configured shipping rate options for the current cart
- added a payment step that reflects the active payment provider
- added a confirmation step that places the order through the existing checkout workflow and renders the resulting order id and total

### Runtime Wiring

- extended the storefront handler with optional checkout dependencies
- wired storefront checkout routes from the same configured checkout service used by `/api/v1/checkout`
- reused the same active shipping provider and active payment provider that are already configured in `cmd/api/main.go`

### Theme

- added four default theme templates for the checkout flow
- added a checkout progress indicator, form layout, summary panel, and confirmation styling to the bundled theme CSS

## Behavior Notes

- the checkout flow currently requires an authenticated customer because the existing order model and checkout service create customer-owned orders
- guests can still maintain carts in the storefront, but checkout pages show a sign-in requirement instead of attempting a guest-order workaround
- shipping choices shown in the storefront reflect the configured shipping provider set passed into the storefront handler
- manual payment works fully server-side in the shipped flow
- Stripe payment pages render provider-aware guidance, but final card confirmation still depends on the existing Stripe provider workflow after the order is created

## Notes

- this PR intentionally layers the storefront flow on top of the existing checkout workflow instead of creating a second checkout implementation
- customer sign-in remains the controlling boundary for order creation until the underlying checkout and order domain contracts support guest checkout explicitly
