## Design Notes

1. **Interface extension**: `SearchEngine` gains `Suggest(ctx, prefix, limit) ([]Suggestion, error)`. Both Postgres and Meilisearch implement it — no runtime type assertions needed.

2. **Suggestion type**: Lightweight `{Text, Type, URL}` struct in the search domain. `Type` is always `"product"` for now; extensible for future category/brand suggestions.

3. **Meilisearch implementation**: Reuses the existing `search()` API method with a minimal `searchRequest` (just Q + Limit, no filters/facets/sort). Maps hits to suggestions via `name` + `slug`. On any Meilisearch error, returns empty list (graceful degradation per spec §3.4).

4. **Postgres implementation**: `ILIKE prefix%` on product `name` column, filtered to `status = 'active'`, ordered by name, limited. Simple and fast for small catalogs.

5. **Limit clamping**: `DefaultSuggestLimit = 5`, `MaxSuggestLimit = 10`. Exported from domain so both engines and the handler can reference them.

6. **Debounce-friendly**: No server-side debounce — the endpoint is stateless and fast. Client-side debounce is the caller's responsibility. Meilisearch prefix search responds in < 50ms for typical indexes.

## Endpoint

```http
GET /api/v1/search/suggest?q=sne&limit=5
```

### Response

```json
{
  "data": {
    "suggestions": [
      { "text": "Sneakers", "type": "product", "url": "/products/sneakers-white" },
      { "text": "Sneaker Care Kit", "type": "product", "url": "/products/sneaker-care-kit" }
    ]
  }
}
```

### Edge cases

| Input              | Behavior                       |
| ------------------ | ------------------------------ |
| `q` empty/missing  | 200 with empty `suggestions`   |
| `limit` invalid    | 422 validation error           |
| `limit` omitted    | Default 5                      |
| Engine unreachable  | Empty list (meili) / error (postgres) |
