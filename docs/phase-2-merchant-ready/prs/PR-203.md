# PR-203: Setup CLI Command

## Summary

New `setup` subcommand that runs first-time setup in a single invocation: verifies database connectivity, applies migrations, runs seeders, and prints a ready-to-use summary.

## Changes

### Modified Files

- **`cmd/api/main.go`**:
  - Added `case "setup"` to the subcommand dispatch switch
  - Added `runSetup(cfg, log)` function implementing the full setup flow
  - Added `setup` entry to `printHelp()` output

### `runSetup` Flow

1. **Parse flags** — `--skip-seed`, `--verbose`, `--non-interactive` (accepted, default behavior)
2. **Database connectivity** — `db.Open()` with built-in `Ping()` validates the connection
3. **Migrations** — `migrate.Run()` applies pending migrations, reports count
4. **Seeders** — Registers `ConfigSeeder`, `AdminSeeder`, `CatalogSeeder` (skipped with `--skip-seed`)
5. **Summary** — Prints store URL, admin API, and docs URL

## Usage

```bash
# Full setup (migrate + seed)
shopanda setup

# Skip seeding
shopanda setup --skip-seed

# Verbose output (includes structured log entries)
shopanda setup --verbose

# All flags
shopanda setup --verbose --skip-seed --non-interactive
```

### Example Output

```text
✓ Database connected
✓ 35 migrations applied
✓ Seed complete (executed: 3, skipped: 0)

Store is ready at http://localhost:8080
Admin API: http://localhost:8080/api/v1/admin
API Docs:  http://localhost:8080/docs
```

## Flags

| Flag                | Effect                                      |
| ------------------- | ------------------------------------------- |
| `--skip-seed`       | Skip the seeding step                       |
| `--verbose`         | Print structured log entries during setup   |
| `--non-interactive` | Accepted for forward compatibility (default)|

## Spec Compliance

| Requirement                          | Status |
| ------------------------------------ | ------ |
| Check Postgres connectivity          | ✅      |
| Run pending migrations               | ✅      |
| Run seeders (idempotent)             | ✅      |
| `--skip-seed` flag                   | ✅      |
| `--verbose` flag                     | ✅      |
| `--non-interactive` flag             | ✅      |
| Print summary with URLs              | ✅      |
| Health endpoint verification         | —¹     |

> **¹** The spec mentions verifying the health endpoint, but `setup` runs
> before the HTTP server starts — there is no server to query. Database
> connectivity (via `db.Open` → `Ping`) serves as the equivalent health
> verification during setup. The `/healthz` endpoint is validated at runtime
> by the Dockerfile `HEALTHCHECK`.

## Tests

No new tests — the function is a thin orchestrator over already-tested
components (`db.Open`, `migrate.Run`, `seed.Registry.Run`). Build and
existing test suite pass cleanly.
