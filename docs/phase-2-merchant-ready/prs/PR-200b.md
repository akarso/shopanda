# PR-200b: Config Hardening — Dotenv Loader + Service-Config Precedence

## Summary

The application now loads `.env` files as an optional development convenience while ensuring real environment variables always take precedence. A startup warning is logged whenever a `.env` file is detected.

## Motivation

Developers need a quick way to configure the app locally, but `.env` files must never silently override config set through the YAML file or shell exports. This change makes the precedence chain explicit and warns operators when a `.env` file is in play.

## Precedence Chain (highest → lowest)

1. **OS environment variables** — shell `export`, or any service manager if used
2. **`.env` file** — development fallback, loaded only if present
3. **YAML config file** — `configs/config.yaml`
4. **Built-in defaults** — hardcoded in `defaults()`

## Changes

### Modified Files

- **`internal/platform/config/config.go`**
  - Added `LoadResult` struct wrapping `*Config` + `DotEnvUsed bool` + `DotEnvPath string`
  - Changed `Load()` signature: returns `*LoadResult, error`
  - `Load()` tries `.env` beside the config file, then falls back to CWD `.env`
  - `loadDotEnv(path string) (bool, error)` — returns proper errors instead of swallowing them; uses `errors.Is(err, fs.ErrNotExist)` to silently skip missing files; checks `scanner.Err()` and `os.Setenv` return values
  - `parseDotEnvLine` doc comment documents intentional limitations (no inline comment stripping, no escape sequences, single quote-pair only)

- **`cmd/api/main.go`**
  - Unpacks `LoadResult` from `config.Load()`
  - Logs `Warn("app.config.dotenv", ...)` with `DotEnvPath` when `.env` is detected

- **`internal/platform/config/config_test.go`**
  - `loadCfg(t, path)` helper uses `t.Chdir` for CWD isolation
  - Added `loadIsolated(t, path)` helper returning `*LoadResult` with CWD isolation
  - All direct `Load(path)` calls replaced with `loadIsolated` or `loadCfg`
  - All `os.WriteFile` calls check errors (`t.Fatalf`)
  - Bare `os.Unsetenv`/`defer os.Unsetenv` replaced with `t.Setenv` + `os.Unsetenv` for safe auto-restore
  - `TestLoad_DotEnvUsedTrue` scopes `SHOPANDA_LOG_LEVEL` via `t.Setenv`; uses `t.Chdir`
  - 7 dotenv-specific tests, all CWD-isolated

- **`.env.example`**
  - Header explains `.env` resolves relative to the config file, with CWD fallback; documents the common `configs/.env` placement
