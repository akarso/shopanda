# PR-225 — Product Listing Page

## Summary

Implemented server-rendered product listing and search pages for the storefront theme:

- added SSR routes for `GET /products` and `GET /search`
- added search-backed listing view models with pagination, sorting, and grid/list state
- added a bundled `product_list.html` template in the default theme
- updated the default theme search form to post into `/search`

No frontend build step. Templates and CSS remain local and server-rendered.

## Scope

### Storefront Routes

- Added `GET /products` for the catalog listing page
- Added `GET /search` for search-result pages driven by `q=...`
- Kept `GET /products/{slug}` for product detail pages
- Storefront SSR routing remains gated by `frontend.enabled`

### Listing Composition

- Extended the storefront HTTP adapter to accept the existing PLP composition pipeline and search engine
- Listing pages now use the search engine for:
  - active-product lookup
  - total hit count
  - offset-based pagination
  - supported sorting (`newest`, `oldest`, `name_asc`, `name_desc`)
- PLP composition still runs over the listing slice so pipeline-added blocks and metadata stay available to the theme layer

### Default Theme

- Added `themes/default/templates/product_list.html`
- Added listing UI states for:
  - empty results
  - sort controls
  - grid/list toggle
  - facet summaries
  - pagination links
- Updated `themes/default/templates/layout.html` to preserve the current search query in the header form
- Updated `themes/default/theme.yaml` so the shared search form posts to `/search`

### Styling

- Extended `themes/default/static/css/style.css` with listing and search page styles
- Added card layouts for both grid and list views
- Reused the existing shell visual language rather than introducing a separate UI system

## Behavior Notes

- Search pages with an empty `q` render an empty prompt instead of executing a blank query
- The storefront preserves `page`, `per_page`, `sort`, `view`, and `q` across pagination and sorting links
- Product cards render indexed price when available; when the active search backend does not supply price data, the card falls back to a product-page CTA instead of showing a wrong price
- Availability text is intentionally conservative because the current Postgres search adapter does not populate explicit stock state

## Files Changed

- `internal/interfaces/http/storefront.go`
  - added SSR listing and search handlers
  - added listing page view models and query parsing
  - wired pagination, sort, and view-state URL generation
- `internal/interfaces/http/storefront_test.go`
  - added listing and search route coverage
  - updated storefront handler construction for PLP and search dependencies
- `cmd/api/main.go`
  - wired `GET /products` and `GET /search`
  - passed PLP pipeline and search engine into the storefront handler
- `themes/default/theme.yaml`
  - updated default storefront search action and nav links
- `themes/default/templates/layout.html`
  - preserved search query state in the shared header form
- `themes/default/templates/product_list.html`
  - added the SSR product listing/search template
- `themes/default/static/css/style.css`
  - added PLP/search styling

## Notes

- The Postgres search adapter still omits explicit price and stock fields from its result payload, so the listing page degrades gracefully instead of fabricating those values.
- This PR keeps listing entirely SSR-first and reuses existing search and composition contracts instead of introducing a separate storefront data path.
