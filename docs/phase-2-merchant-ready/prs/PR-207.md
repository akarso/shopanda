# PR-207 — Stripe provider plugin

## Goal

Implement the Stripe `PaymentProvider` with PaymentIntent creation and
client-secret flow, enabling production payment processing as an opt-in
alternative to the built-in manual provider.

## Scope

| Area | Change |
|---|---|
| `internal/domain/payment/payment.go` | Added `MethodStripe PaymentMethod = "stripe"` and updated `IsValid()` |
| `internal/domain/payment/provider.go` | Extended `ProviderResult` with `Pending bool` and `ClientSecret string` for async provider flows |
| `internal/application/checkout/initiate_payment_step.go` | Added `result.Pending` branch — validates non-empty `ProviderRef`, payment stays pending, `client_secret` set in checkout context |
| `internal/interfaces/http/checkout.go` | Checkout response now includes `payment.client_secret` when present in context meta |
| `internal/infrastructure/stripepay/provider.go` | New `Provider` implementing `payment.Provider` via Stripe PaymentIntents HTTP API; `NewProvider` returns `(*Provider, error)`; 30 s HTTP timeout; `Idempotency-Key` header; response truncation detection; validates `id`/`client_secret` non-empty |
| `internal/platform/config/config.go` | Added `PaymentConfig.Stripe` (`enabled`, `secret_key`) with env var overlay; `strconv.ParseBool` for truthy parsing |
| `cmd/api/main.go` | Conditional provider selection: Stripe when enabled + `SHOPANDA_PAYMENT_STRIPE_SECRET_KEY` env var set (YAML `secret_key` ignored with warning); documented that Stripe webhook adapter is PR-208 scope |
| `internal/interfaces/http/payment_webhook_test.go` | Updated `TestPaymentWebhook_UnknownProvider` to use `"unknown-provider"` since `"stripe"` is now a valid method |

## Design notes

1. **No SDK dependency** — The Stripe provider uses plain `net/http` with
   form-encoded requests to the PaymentIntents API. This avoids pulling in
   the `stripe-go` SDK and its transitive dependencies, keeping `go.mod`
   minimal. The HTTP client is testable via `WithBaseURL` option.

2. **Pending provider result** — `ProviderResult.Pending` indicates the
   payment awaits external confirmation (webhook). The checkout step leaves
   the payment in `StatusPending`, stores the `ProviderRef` (PaymentIntent ID),
   and passes `client_secret` through checkout context for the HTTP response.
   Existing synchronous providers (`manual`) are unaffected — they never set
   `Pending=true`.

3. **Config-driven provider selection** — `payment.stripe.enabled` +
   `SHOPANDA_PAYMENT_STRIPE_SECRET_KEY` env var. The Stripe secret key is
   sourced exclusively from the environment variable; YAML `secret_key`
   values are ignored with a warning to prevent secrets in config files.
   When the env var is empty, the system falls back to the manual provider.

4. **Checkout response extension** — The checkout handler now returns
   `{ "order": {...}, "payment": { "client_secret": "pi_..." } }` when a
   client secret is present. Frontends use this with Stripe.js to confirm
   the payment. Non-Stripe checkouts return only `{ "order": {...} }`.

5. **Amount already in minor units** — `shared.Money.Amount()` returns cents
   (smallest currency unit), which Stripe expects. No conversion needed.

6. **Stripe webhook adapter is PR-208** — The current `PaymentWebhookHandler`
   uses HMAC-SHA256 and a plain `{payment_id, provider_ref, success}` envelope.
   Stripe requires a dedicated adapter that verifies `Stripe-Signature` headers
   and parses Stripe events. Until PR-208, Stripe-initiated payments remain in
   `StatusPending` after checkout.

7. **Pending result invariants** — `InitiatePaymentStep` rejects pending
   results with empty `ProviderRef` to prevent unreconcilable payments.
   `NewProvider` returns an error instead of panicking on empty secret key.
   The provider validates that the Stripe response contains non-empty `id`
   and `client_secret` before returning `ProviderResult`.

## Tests

9 tests in `stripepay/provider_test.go` (all new):

- `TestProvider_Method` — returns `MethodStripe`
- `TestProvider_Initiate_Success` — verifies request format (incl. `Idempotency-Key`), parses response, checks `Pending`/`ClientSecret`
- `TestProvider_Initiate_APIError` — Stripe 400 with error message
- `TestProvider_Initiate_APIErrorNoMessage` — Stripe 500 without structured error
- `TestProvider_Initiate_NilPayment` — nil guard
- `TestProvider_Initiate_NetworkError` — deterministically closed httptest server
- `TestProvider_Initiate_InvalidJSON` — malformed response body
- `TestNewProvider_EmptyKey` — returns error for empty secret key
- `TestProvider_Initiate_MissingFields` — rejects response missing `client_secret`

3 tests in `checkout/payment_shipping_steps_test.go` (new):

- `TestInitiatePaymentStep_Pending` — verifies pending flow: payment stays pending, ProviderRef stored, client_secret in meta
- `TestInitiatePaymentStep_Pending_NoClientSecret` — no `client_secret` meta when `ClientSecret` is empty
- `TestInitiatePaymentStep_Pending_EmptyProviderRef` — rejects pending result with empty ProviderRef

1 test in `payment/payment_test.go` (updated):

- `TestPaymentMethod_IsValid` — added `MethodStripe` case
