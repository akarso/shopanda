# PR-223 — Settings Page

## Summary

Implemented the admin settings page for the embedded SPA:

- Settings page at `/admin/settings`
- Grouped settings sections for store, email, media, currency, and tax
- Backend DB-backed config API for grouped reads and writes
- SMTP test endpoint for validating email delivery settings
- Integration with the existing admin store API for core store fields

No build step. Vanilla JS only.

## Scope

### Settings Page

- Renders grouped sections for:
  - Store Info
  - Email
  - Media
  - Currency
  - Tax
- Loads existing values on page load and saves each section independently
- Uses the embedded admin SPA routing at `/admin/settings`

### Config API

- Reads grouped config values from: `GET /api/v1/admin/config?group={group}`
- Persists config values to the DB config layer with: `PUT /api/v1/admin/config`
- Supports these config groups:
  - `store`
  - `email`
  - `media`
  - `currency`
  - `tax`

### Store Info

- Uses the existing admin store API for core store entity fields:
  - code
  - name
  - domain
  - country
  - language
  - currency
  - default flag
- Uses the grouped config API for store presentation fields:
  - `store.address`
  - `store.logo`
- If no store exists yet, the page creates one through `POST /api/v1/admin/stores`

### Email Settings

- Saves SMTP settings through the config API:
  - `mail.smtp.host`
  - `mail.smtp.port`
  - `mail.smtp.user`
  - `mail.smtp.password`
  - `mail.smtp.from`
- Sends a test message with: `POST /api/v1/admin/config/test-email`
- Test-email requests can use current form values immediately, without requiring a prior save
- Secret values are redacted in config reads and are not echoed back in plaintext by config writes

### Media, Currency, and Tax

- Media settings include storage mode and local/S3 fields
- Currency settings include default currency and display format
- Tax settings include default tax class and included/excluded pricing flag

## Auth & Permissions

Admin settings endpoints require authenticated admin access and settings permissions:

- `GET /api/v1/admin/config` -> `rbac.SettingsRead`
- `PUT /api/v1/admin/config` -> `rbac.SettingsWrite`
- `POST /api/v1/admin/config/test-email` -> `rbac.SettingsWrite`

Store entity operations continue to use store permissions through the existing admin store routes.

Common failures:

- `401 Unauthorized` for missing or invalid authentication
- `403 Forbidden` for authenticated users without required permission
- `422 Unprocessable Entity` for invalid config keys, empty payloads, or invalid SMTP test input

## Notes

- Grouped config reads merge DB values over runtime config defaults so the settings page can render meaningful values before anything is persisted.
- Config writes now use a bulk repository operation so grouped updates succeed or fail as a single unit.
- The current store section uses the first default store, or the first available store when no default exists.
- The store section intentionally reuses the existing store CRUD API instead of duplicating store entity persistence inside the config layer.
- The SMTP password is treated as write-only in the admin API: reads return a redacted sentinel and unchanged form submissions do not overwrite the stored secret.
