# PR-205 — Core Email Templates

## Summary

Adds three responsive HTML email templates (order confirmation, password reset,
order shipped) and wires them into the notification service via new event
handlers. Templates use `mail.EmailData` for consistent store-level and
per-template data access.

## Templates Created

### `templates/emails/order_confirmed.html`

- **Trigger:** `order.paid` event
- **Subject:** `Order {{.Data.OrderID}} — Confirmation`
- **Content:** Thank-you message, order items table (product / qty / price),
  total, "View Order" CTA button
- **Data:** `OrderID`, `FirstName`, `Items` (array of `{Name, Qty, Price}`),
  `Total`

### `templates/emails/password_reset.html`

- **Trigger:** `customer.password_reset.requested` event
- **Subject:** `Reset your password`
- **Content:** Reset instructions, CTA button with token URL, expiry notice,
  "if you didn't request this" security note
- **Data:** `FirstName`, `ResetURL`, `ExpiresIn`

### `templates/emails/order_shipped.html`

- **Trigger:** `shipment.shipped` event
- **Subject:** `Order {{.Data.OrderID}} — Shipped`
- **Content:** Shipping confirmation, tracking number + carrier (conditional),
  "View Order" CTA button
- **Data:** `OrderID`, `FirstName`, `TrackingNumber` (optional), `Carrier`
  (optional)

## Architecture

```text
Event Bus
  ├─ order.paid                      → HandleOrderPaid         → order_confirmed template
  ├─ customer.password_reset.requested → HandlePasswordReset   → password_reset template
  └─ shipment.shipped                → HandleShipmentShipped    → order_shipped template
                                           ↓
                                     enqueueEmail()
                                           ↓
                                     email.send job → worker → SMTP
```

### Data Flow

All handlers now pass `mail.EmailData`:

```go
ed := mail.EmailData{
    StoreURL: s.storeURL,           // from WithStoreURL option
    Data: map[string]interface{}{    // template-specific
        "OrderID":   "...",
        "FirstName": "...",
    },
}
msg, err := s.templates.Render("order_confirmed", cust.Email, ed)
```

Templates access store fields via `{{.StoreURL}}` and per-template data via
`{{.Data.OrderID}}`.

## Key Design Decisions

1. **Template naming uses underscores** — `order_confirmed` (not `order.confirmed`)
   matches the filename convention (`order_confirmed.html`). Existing hardcoded
   template renamed accordingly.

2. **Fallback templates in `RegisterTemplates`** — built-in templates use the
   same `{{.Data.xxx}}` shape as file templates. `LoadDir` overwrites them when
   files exist, providing a seamless file → hardcoded fallback chain.

3. **Service uses Option pattern** — `WithResetBaseURL` and `WithStoreURL`
   configure URL generation without growing the constructor signature.

4. **`enqueueEmail` helper** — DRYs the job-creation + enqueue + logging pattern
   shared by all three handlers.

5. **Shipping repo added as required dep** — `HandleShipmentShipped` needs order
   lookups (shipment event carries `OrderID`), so the repo is required at
   construction time.
