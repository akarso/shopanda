# PR-229 — Account Pages

## Summary

Implemented the storefront account slice as server-rendered pages:

- added login and registration pages at `/account/login` and `/account/register`
- added authenticated account pages for order history, order detail, and profile management
- introduced storefront session-cookie auth so SSR account pages can persist browser sign-in state
- reused the existing auth service, order repository, and account deletion service instead of creating a parallel storefront-only auth stack

## Scope

### Storefront Account Pages

- added login and registration forms with server-side validation and session-cookie sign-in
- added `/account/orders` for customer order history and `/account/orders/{orderId}` for order detail
- added `/account/profile` for profile edits, password changes, and account deletion/logout actions

### Application Reuse

- extended the auth application service with profile update and password change methods
- reused the existing order repository for order history rendering
- reused the existing account deletion service for the storefront delete-account action

### Session and Form Security

- extended the HTTP auth middleware to accept the storefront session cookie when no bearer token is present
- expanded storefront CSRF protection from checkout-only routes to account form routes as well
- kept SSR account flows on regular HTML forms and redirects with no frontend build step

### Theme

- added five default theme templates for login, registration, orders, order detail, and profile pages
- added account page styling that matches the existing bundled storefront visual language

## Behavior Notes

- storefront login and registration redirect authenticated customers to `/account/orders`
- account pages redirect guests to the login page and preserve a safe local redirect target
- password changes invalidate prior tokens and clear the storefront session cookie so the customer signs in again
- account deletion reuses the existing deletion workflow and clears the storefront session cookie after completion

## Notes

- this PR keeps browser auth state at the HTTP interface layer by reusing JWTs in an HttpOnly storefront session cookie rather than introducing a second session model
- storefront account actions remain thin adapters over existing application/domain services so account behavior stays consistent between JSON APIs and SSR pages
- renaming the CSRF cookie from `shopanda_checkout_csrf` to `shopanda_csrf` can produce one transient browser-side `403 forbidden` after deploy while older forms or cookies are still in circulation; a browser restart or waiting for the stale page/session to expire clears the mismatch
