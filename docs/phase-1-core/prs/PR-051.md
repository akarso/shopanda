# PR-051 — Category HTTP Handlers

## Summary

Public read endpoints for categories: full tree, single category, and products-by-category with pagination.

## Changes

### Domain — Repository Interfaces

- **`CategoryRepository.FindAll(ctx) ([]Category, error)`** — returns all categories ordered by position/name, used to build the tree in one query.
- **`ProductRepository.FindByCategoryID(ctx, categoryID, offset, limit) ([]Product, error)`** — returns paginated products via the `product_categories` join table.

### Infrastructure — Postgres Implementations

- **`CategoryRepo.FindAll`** — single `SELECT` ordered by position asc, name asc.
- **`ProductRepo.FindByCategoryID`** — `INNER JOIN product_categories`, respects tx context, validates inputs, caps limit at 100.

### Interfaces — HTTP Handler

- **`internal/interfaces/http/category.go`** — `CategoryHandler` with three endpoints:
  - `Tree()` → `GET /api/v1/categories` — loads all categories, builds nested tree in memory via two-pass algorithm (create nodes → attach children).
  - `Get()` → `GET /api/v1/categories/{id}` — returns single category by ID.
  - `Products()` → `GET /api/v1/categories/{id}/products` — verifies category exists, then returns paginated products. Reuses `parsePagination()` helper.

### Wiring — `cmd/api/main.go`

- Instantiates `CategoryRepo`, `CategoryHandler`.
- Registers 3 public routes under `/api/v1/categories`.

### Test Fixes

- Added `FindByCategoryID` stub to 4 mock `ProductRepository` implementations in existing test files.

## Routes

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| GET | `/api/v1/categories` | Public | Full category tree |
| GET | `/api/v1/categories/{id}` | Public | Single category |
| GET | `/api/v1/categories/{id}/products` | Public | Paginated products in category |
