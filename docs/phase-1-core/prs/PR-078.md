# PR-078 — Seed Framework

## Summary

Adds the seed data framework: `Seeder` interface, `Registry` for ordered execution, `SeederFunc` adapter, and `app seed` CLI command. No actual seed data is included — that comes in PR-079 (core seed data).

## Design Decisions

1. **Interface-based** — `Seeder` interface with `Name()` and `Seed(ctx, deps)` so both struct-based and function-based seeders work
2. **Registry pattern** — `NewRegistry()` + `Register()` + `Run()` matches the plugin registration model; seeders execute in registration order
3. **Deps struct** — passes `*sql.DB` and `logger.Logger` so seeders can create any repository they need without the framework knowing which repos exist
4. **Idempotency is the seeder's responsibility** — per the spec, each seeder checks for existing data before inserting (using stable identifiers like email or slug)
5. **Sequential execution** — seeders may depend on each other (e.g. products need categories), so ordering matters and first error stops the run
6. **SeederFunc adapter** — `NewSeederFunc(name, fn)` for simple seeders that don't need a full struct
7. **Duplicate name panic** — catches wiring bugs at startup, not at runtime
8. **No environment modes yet** — `--env=dev/prod` filtering is deferred until PR-079 when actual seed data exists

## Test Coverage (7 tests)

| Test | Validates |
|------|-----------|
| `TestRegistry_RunEmpty` | Zero seeders → zero executed, no error |
| `TestRegistry_RunOrder` | 3 seeders execute in registration order |
| `TestRegistry_RunStopsOnError` | Failing seeder stops execution, error propagated, subsequent seeders skipped |
| `TestRegistry_DuplicateNamePanics` | Duplicate seeder name panics at registration |
| `TestRegistry_EmptyNamePanics` | Empty seeder name panics at registration |
| `TestSeederFunc` | Function adapter: name returned correctly, function invoked |
| `TestRegistry_DepsPassed` | Deps struct passed through to seeder |
