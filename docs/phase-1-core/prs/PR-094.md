# PR-094: Store Domain + Resolution

## Summary

Adds the `Store` entity, repository, migration, domain-based resolution middleware, and admin CRUD endpoints. This is the first piece of multi-store support.

## Design

### Store entity

- Fields: `ID`, `Code`, `Name`, `Currency` (3-letter ISO), `Country` (2-letter ISO), `Domain`, `IsDefault`, `CreatedAt`, `UpdatedAt`
- `NewStore` validates all required fields, rejects invalid currency/country lengths, uppercases currency and country
- `NewStoreFromDB` reconstructs from database without validation

### Migration

- `stores` table with `id` PK, `code` UNIQUE
- Partial unique index on `domain` (`WHERE domain != ''`) — allows multiple stores without domains
- Partial unique index on `is_default` (`WHERE is_default = TRUE`) — enforces at most one default store

### Store resolution middleware

- Extracts domain from `r.Host` (strips port)
- Looks up store by domain via `FindByDomain`
- On miss or error, falls back to default store via `FindDefault`
- Injects resolved store into request context via `store.WithStore`
- Placed after `AuthMiddleware` in the middleware chain

### Admin endpoints

| Method | Path | Handler |
|--------|------|---------|
| GET | `/api/v1/admin/stores` | `List()` — returns all stores |
| POST | `/api/v1/admin/stores` | `Create()` — validates and creates a store |
| PUT | `/api/v1/admin/stores/{id}` | `Update()` — partial update with field-level validation |

- `store.created` and `store.updated` events published via event bus
