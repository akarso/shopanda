# PR-012 — Composition Pipeline (Typed)

## Summary

Introduces the generic composition pipeline: `Pipeline[T]`, `Step[T]` interface, `Block` type, and typed contexts for product pages (PDP) and product listings (PLP).

## Files

| File | Purpose |
|------|---------|
| `internal/application/composition/step.go` | `Step[T]` generic interface — `Name() string`, `Apply(ctx *T) error` |
| `internal/application/composition/pipeline.go` | `Pipeline[T]` — ordered step execution with error propagation |
| `internal/application/composition/block.go` | `Block` struct — UI-agnostic component (`Type` + `Data`) |
| `internal/application/composition/product_context.go` | `ProductContext` (PDP) — product, currency, country, blocks, meta |
| `internal/application/composition/listing_context.go` | `ListingContext` (PLP) — products, filters, sort options, blocks, meta |
| `internal/application/composition/pipeline_test.go` | 8 tests covering both pipeline types |

## Design Decisions

- **Go generics**: `Pipeline[T]` and `Step[T]` use type parameters — compile-time safety, no runtime casting.
- **Sequential execution**: Steps run in insertion order. First error stops the pipeline.
- **Error wrapping**: Pipeline wraps step errors with the step name for debuggability.
- **No position-based registration**: `AddStep` is append-only. Position-based insertion (`before:`, `after:`) is deferred to PR-049 (plugin registration API).
- **Separate context constructors**: `NewProductContext` and `NewListingContext` initialize all slices/maps to avoid nil checks downstream.
- **Filter and SortOption types**: Defined alongside `ListingContext` — minimal structs for listing facets.

## Tests

8 tests covering:
- Empty pipeline execution (no-op)
- Single step adds block
- Multiple steps execute in order
- Pipeline stops on first error
- `Steps()` returns registered step names
- Listing pipeline with blocks + filters
- `NewProductContext` initializes all fields
- `NewListingContext` initializes all fields
