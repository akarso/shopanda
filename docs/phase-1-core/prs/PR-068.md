# PR-068 — Theme Engine (Basic SSR)

## Summary

Introduces the theme engine — template loading with layout support and a `Render()` function for server-side rendering. This is the foundation layer; HTTP wiring and config integration follow in later PRs.

## Design Decisions

### Per-page template parsing

Go's `html/template.ParseFiles` merges all `define` blocks into a single namespace. If both `product.html` and `listing.html` define `"title"` and `"content"`, the last file parsed wins. To solve this, the engine parses each page template separately alongside `layout.html`, creating isolated template sets per page. This allows multiple pages to define the same block names without conflict.

### Layout is optional

Themes without `layout.html` work fine — each page template renders standalone. This supports both full-layout themes and simple snippet-based themes.

### Domain layer placement

The engine lives in `internal/domain/theme/` per the spec: it's a pure domain concept with no infrastructure dependencies (uses only `html/template`, `os`, `gopkg.in/yaml.v3`).

## Template Convention

```
themes/<name>/
  theme.yaml           # name + version
  templates/
    layout.html        # optional layout with {{ template "title" . }} and {{ template "content" . }}
    product.html       # defines "title" and "content" blocks, invokes layout
    listing.html       # same pattern
```

Page templates define blocks and invoke the layout:
```html
{{ define "title" }}{{ .Name }}{{ end }}
{{ define "content" }}<h1>{{ .Name }}</h1>{{ end }}
{{ template "layout.html" . }}
```
