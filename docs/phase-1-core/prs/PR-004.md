# PR-004: Request context + correlation IDs

## Phase
Phase 0 — Project Scaffold

## Scope
`RequestID` middleware, correlation ID propagation, context-aware logger

## Files Created
- `internal/platform/requestctx/requestctx.go` — `WithRequestID`/`RequestID` context helpers, `Middleware` (generates or echoes `X-Request-ID` header), crypto/rand ID generation
- `internal/platform/requestctx/requestctx_test.go` — 5 tests: context round-trip, empty context, middleware generates ID, middleware echoes incoming header, uniqueness across 10 requests
- `internal/platform/logger/context.go` — `ContextLogger` wrapping `Logger` interface, `WithContext()` constructor, auto-merges `request_id` from context into every log entry
- `internal/platform/logger/context_test.go` — 6 tests: request_id inclusion, absent request_id, nil caller context, warn/error levels, interface compliance

## Key Decisions
- `requestctx` is a separate package from `logger` to avoid circular deps — any package can read request_id without importing logger
- `ContextLogger` satisfies `Logger` interface — can be passed anywhere a `Logger` is accepted
- Middleware echoes `X-Request-ID` from caller if present (load balancer, API gateway), generates new one otherwise
- No main.go wiring yet — middleware will be plugged into HTTP server in PR-007
- `job_id` correlation deferred to Phase 15 (Jobs)

## Usage Pattern
```go
// In HTTP handler (after PR-007 wires middleware):
log := logger.WithContext(baseLog, r.Context())
log.Info("order.created", map[string]interface{}{"order_id": "..."})
// Output includes "request_id" automatically
```
