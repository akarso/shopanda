# PR-044 — Payment Webhook Endpoint

## Summary

Adds `POST /api/v1/payments/webhook/{provider}` — a public HTTP endpoint that
receives payment result notifications from external (or internal) payment
providers and transitions the payment status accordingly.

## Design

### Route

```text
POST /api/v1/payments/webhook/{provider}
```

**Public** (no auth middleware) — called by external payment providers.

### Request Body

```json
{
  "payment_id": "pay-abc",
  "provider_ref": "ref-from-provider",
  "success": true
}
```

### Flow

1. Extract `{provider}` path parameter, validate it as a known `PaymentMethod`
2. Decode JSON body, validate `payment_id` is present
3. Look up payment by ID; 404 if not found
4. Verify payment method matches the provider in the URL
5. Capture `prevUpdatedAt` for optimistic locking
6. Apply `Complete(providerRef)` or `Fail()` based on `success` flag
7. Persist via `UpdateStatus(ctx, p, prevUpdatedAt)` — conflict on stale writes
8. Publish `payment.completed` or `payment.failed` domain event
9. Return `200 {"data": {"status": "accepted"}}`

### Error Responses

| Condition | HTTP Status | Code |
|-----------|-------------|------|
| Unknown provider | 422 | validation |
| Invalid/missing body | 422 | validation |
| Payment not found | 404 | not_found |
| Provider mismatch | 422 | validation |
| Non-pending payment | 422 | validation |
| Concurrent update | 409 | conflict |
| Repo/system error | 500 | internal |

## Tests

8 tests in `payment_webhook_test.go`:

- `TestPaymentWebhook_Success` — complete a pending payment
- `TestPaymentWebhook_Failure` — fail a pending payment
- `TestPaymentWebhook_NotFound` — payment does not exist
- `TestPaymentWebhook_UnknownProvider` — unknown provider rejected
- `TestPaymentWebhook_InvalidBody` — malformed JSON
- `TestPaymentWebhook_MissingPaymentID` — empty payment_id
- `TestPaymentWebhook_AlreadyCompleted` — non-pending payment rejected
- `TestPaymentWebhook_RepoError` — repository error propagates as 500
