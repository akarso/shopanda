# PR-098 — Cookie Consent + GDPR

## Summary

Adds a cookie consent model and GDPR data-access endpoints so customers can
manage their tracking preferences and exercise data-portability / right-to-erasure
rights.

## New endpoints

| Method | Path | Auth | Description |
|--------|------|------|-------------|
| `GET` | `/api/v1/account/consent` | required | Return current consent preferences (defaults if none saved) |
| `PUT` | `/api/v1/account/consent` | required | Update analytics / marketing preferences |
| `GET` | `/api/v1/account/data` | required | Return all personal data (profile + orders + consent) |
| `GET` | `/api/v1/account/export` | required | Same as `/data` with `Content-Disposition: attachment` |
| `DELETE` | `/api/v1/account` | required | Delete consent + customer; emit `customer.deleted` event |

## Design decisions

* **Necessary consent is always true** — the spec mandates session cookies cannot be rejected.
* **Upsert pattern** — first `PUT` creates the consent row; no separate create endpoint needed.
* **Export reuses GetData handler** — `Export()` delegates to `GetData()` with an added `Content-Disposition` header to avoid duplication.
* **`customer.deleted` event** — published after deletion so listeners (e.g. order anonymisation, cart cleanup) can react asynchronously.
* **`ON DELETE CASCADE` on FK** — consent rows are automatically removed when the customer row is deleted, providing a safety net even if the handler is bypassed.
